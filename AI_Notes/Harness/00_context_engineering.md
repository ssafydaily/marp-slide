# Context Engineering for Coding Agents

- [원문: Context Engineering for Coding Agents](https://martinfowler.com/articles/exploring-gen-ai/context-engineering-coding-agents.html)


- 컨텍스트 엔지니어링은 모든 유형의 에이전트와 LLM 사용에 있어 매우 중요 
  > *바라 니 수브라마니암*:
  > "모델이 보는 정보를 선별하여 더 나은 결과를 얻도록 하는 것"
  

- 코딩 에이전트 분야에서는 컨텍스트 엔지니어링 접근 방식과 용어가 새롭게 부상
- 그 기반은 도구에서 제공하는 구성 기능(예: "규칙(rules)", "기술(skills)")이며, 
- 세부적인 내용은 이러한 기능을 개념적으로 어떻게 활용하는지("사양(specs)", 다양한 워크플로)에 대한 것

# 재사용 가능한 프롬프트

- 거의 모든 형태의 AI 코딩 컨텍스트 엔지니어링은 궁극적으로 프롬프트가 포함된 여러 개의 마크다운 파일을 사용한다
-프롬프트는 LLM게 보내는 텍스트: 프롬프트의 의도는 크게 두 가지 범주로 구분

> - **Instruction**: 에이전트에게 특정 작업을 수행하도록 지시하는 프롬프트(예: "다음과 같은 방식으로 E2E 테스트를 작성하세요....")

> - **Guidance** (규칙, 가이드라인이라고도 함): 에이전트가 따라야 할 일반적인 관례, 예: "항상 서로 독립적인 테스트를 작성하십시오."

- 이들은 종종 서로 겹치지만, 이들을 구분하는 것이 유용하다

----------------
# 컨테스트 인터페이스

- LLM이 필요에 따라 더 많은 컨텍스트 정보를 얻는 방법
  - 명확히 부르는 용어가 없다. 저자가 명명

> - **Tools** : bash 명령어 호출, 파일 검색 등과 같은 내장 기능.

> - **MCP Servers**: 사용자 지정 프로그램 또는 스크립트로, 사용자의 컴퓨터(또는 서버)에서 실행되어 에이전트가 데이터 소스 및 기타 작업에 접근 가능하게 함

> - **Skills** : 컨텍스트 엔지니어링에 새롭게 추가된 것으로 LLM이 현재 작업에 관련성이 있다고 판단될 때 필요에 따라 로드할 수 있는 추가 리소스, 지침, 문서, 스크립트 등

- 이런 설정을 많이 할수록 컨텍스트에서 차지하는 공간이 늘어남 
- 특정 작업에 필요한 컨텍스트 인터페이스가 무엇인지 전략적으로 고려하는 것이 현명

# 워크스페이스 파일

- 코딩 에이전트에서 가장 기본적이면서도 강력한 컨텍스트 인터페이스는 파일 읽기 및 검색
- 현재 코드베이스를 이해하는 데 필요함

> 기존 코드가 컨텍스트 역할을 얼마나 잘 수행하는지, 즉 AI 친화적인 코드베이스 설계를 하고 있는지 생각해 볼 가치가 있다.

----------

# 컨텍스트 로드 여부 및 시점: 누가 컨텍스트 로드를 결정하는가?

- **LLM(로딩 로직 관리자)**: 
  - LLM이 로드 시점을 스스로 결정. 비지도 학습 방식으로 에이전트를 실행에 필수적
  - 하지만, LLM이 예상대로 컨텍스트를 로드할지 여부는 항상 불확실성(혹은 비결정성)이 존재 예: 스킬

- **인간 개입**: 
  - 인간이 상황에 맞춰 개입해서 제어권 확보 가능
  - 그러나, 전반적인 자동화 수준은 저하 예: 슬래시 명령어

- **에이전트 소프트웨어**: 
  - 일부 컨텍스트 기능은 에이전트 소프트웨어 자체에서 특정 시점에 트리거 예: Claude 코드 후킹

----------------------
# 얼마나: 맥락을 최대한 좁게 유지하는 것

- 컨텍스트 엔지니어링의 목표 중 하나는 제공되는 컨텍스트의 양을 적절하게 조절하는 것
- 컨텍스트 윈도우가 기술적으로 매우 커졌다고 정보를 무분별하게 쏟아붓는 것은 좋지 않다. 
- 너무 많은 컨텍스트는 에이전트 효율성 저하 및 과도한 비용 

- 크기 관리의 일부는 개발자에게 달려 있다

> 가이드라인
> - 규칙 파일과 같은 컨텍스트를 점진적으로 구축하고 처음부터 너무 많은 내용을 넣지 않는다.
> - 모델이 상당히 강력해졌기 때문에 6개월 전에 컨텍스트에 넣어야 했던 내용이 이제는 더 이상 필요하지 않을 수 있다

#### `claude.md`
내용: 안내
로딩을 결정하는 주체: 클로드 코드 - 세션 시작 시 항상 사용됨
사용 시점: 프로젝트 전체에 적용되는 가장 자주 반복되는 일반적인 규칙에 사용합니다.
사용 사례 예시:
- "저희는 npm 대신 yarn을 사용합니다."
- "무엇을 실행하기 전에 가상 환경을 활성화하는 것을 잊지 마세요."
- "리팩토링을 할 때는 하위 호환성을 신경 쓰지 않습니다."

다른 코딩 도우미: 기본적으로 모든 코딩 도우미는 "규칙 파일"이라는 기본 기능을 가지고 있으며, 이를 표준화하려는 시도가 있습니다. [AGENTS.md](https://agents.md/)

#### Rules
<hr>
내용: 안내

파일 로드를 결정하는 주체: 클로드 코드(Claude Code), 설정된 경로의 파일들이 모두 로드되면 로드를 결정합니다.

사용 시점: 가이드라인을 체계화하고 모듈화하여 항상 로드되는 CLAUDE.md 파일의 크기를 줄이는 데 도움이 됩니다. 규칙은 파일(예: 모든 TypeScript 파일의 경우 *.ts)에 적용 범위를 지정할 수 있으며, 이렇게 하면 관련성이 있을 때만 로드됩니다.

예시 사용 사례: "Bash 스크립트를 작성할 때 변수는 $var가 아닌 ${var}와 같이 참조해야 합니다." 경로: **/*.sh

다른 코딩 도우미: 점점 더 많은 코딩 도우미들이 경로 기반 규칙 설정을 지원하고 있습니다. 예를 들어 GH Copilot과 Cursor가 있습니다.

# 슬래시 명령어

내용: 지침

로딩을 결정하는 주체: 사람

사용 시점: 특정하고 자세한 프롬프트가 필요한 일반적인 작업(검토, 커밋, 테스트 등)을 메인 컨텍스트 내에서 직접 실행하려는 경우. Claude Code에서는 더 이상 사용되지 않으며, Skills로 대체되었습니다.

사용 사례 예시: /code-review · /e2e-test·/prep-commit

기타 코딩 도우미: 일반적인 기능 (예: GH Copilot 및 Cursor)

# Skills

내용: 지침, 설명서, 문서, 스크립트 등
누가 로딩을 결정할 것인가: LLM(기술 설명 기준) 또는 사람
사용 시점: 가장 간단하게는, 현재 작업에 필요한 경우에만 "지연 로딩"하려는 안내 또는 지침에 사용합니다. 하지만 스킬 폴더에 원하는 추가 리소스와 스크립트를 넣고, 메인 파일에서 참조하여 SKILL.md로드할 수도 있습니다.

사용 사례 예시:
- JIRA 접근 권한 (예: 상담원이 CLI를 사용하여 JIRA에 접근하는 방법을 설명하는 스킬)
- React 컴포넌트 개발을 위한 규칙
- “XYZ API를 통합하는 방법”

다른 코딩 도우미: Cursor의 "지능적으로 적용" 규칙은 항상 이와 비슷했지만, 이제는 Claude Code 스타일의 스킬로 전환하고 있습니다.

# 하위 에이전트
내용: 지침 + 모델 및 사용 가능한 도구 세트 구성; 자체 컨텍스트 창에서 실행되며 병렬 처리가 가능합니다.

누가 로딩을 결정하는가: LLM인가, 아니면 사람인가?

사용 시점:
- 효율성을 높이거나(더욱 의도적인 맥락을 통해 결과를 개선하거나) 비용을 절감하기 위해, 별도의 맥락에서 실행하는 것이 적합하고 가치 있는 일반적인 대규모 작업들.
- 일반적으로 기본 모델이 아닌 다른 모델을 사용하고 싶은 작업
- 특정 도구나 MCP 서버가 필요하지만 기본 컨텍스트에서 항상 사용 가능하도록 해두고 싶지 않은 작업
- 오케스트레이션 가능한 워크플로

사용 사례 예시:
- 방금 구축한 모든 항목에 대한 엔드투엔드 테스트를 생성하세요.
- 기존 세션의 영향을 받지 않고, 별도의 환경과 다른 모델을 사용하여 코드 리뷰를 진행함으로써 "제2의 의견"을 얻을 수 있습니다.
- 서브에이전트는 클로드플로우나 가스타운과 같은 군집 실험의 기본 요소입니다.

-----------
# MCP 서버
내용: 에이전트가 모델 컨텍스트 프로토콜(MCP)을 통해 데이터 소스 및 기타 작업에 접근할 수 있도록 해주는 프로그램으로, 사용자의 컴퓨터(또는 서버)에서 실행됩니다.

로딩을 결정하는 사람: LLM

사용 시점: 에이전트에게 API 또는 컴퓨터에서 실행되는 도구에 대한 액세스 권한을 부여하려는 경우에 사용합니다. 마치 컴퓨터에 여러 옵션이 있는 스크립트가 있고, 이러한 옵션이 구조화된 방식으로 에이전트에 노출되는 것과 같다고 생각하면 됩니다. LLM이 이 기능을 호출하기로 결정하면, 도구 호출 자체는 일반적으로 예측 가능한 작업입니다. 최근에는 스크립트 및 CLI 사용 방법을 설명하는 스킬로 일부 MCP 서버 기능을 대체하는 추세입니다.

예시 사용 사례: JIRA 접근(Atlassian API 호출을 실행할 수 있는 MCP 서버) · 브라우저 탐색(예: Playwright MCP) · 사용자의 컴퓨터에 있는 지식 기반 접근

기타 코딩 도우미: 현재 대부분의 일반적인 코딩 도우미는 MCP 서버를 지원합니다.

# hooks
내용: 스크립트

누가 로드할지 결정하는가: Claude 코드 수명 주기 이벤트

사용 시점: 파일을 편집하거나, 명령을 실행하거나, MCP 서버를 호출하는 등의 작업을 할 때마다 특정 결과가 반드시 발생하도록 하고 싶을 때 사용합니다 .

사용 사례 예시:

- 사용자 지정 알림
- 파일을 편집할 때마다 해당 파일이 자바스크립트 파일인지 확인하고, 자바스크립트 파일인 경우 Prettier를 실행하세요.
- 클로드 코드의 관찰 가능성 활용 사례, 예를 들어 실행된 모든 명령어를 어딘가에 기록하는 것 등이 있습니다.

다른 코딩 보조 기능: 훅(Hooks)은 아직까지는 흔하지 않은 기능입니다. Cursor에서 이제 막 훅 지원을 시작했습니다.

# 컨테스트 구성 공유
공유의 어려움:

- 정보를 공유하는 사람과 받는 사람의 상황이 최대한 비슷해야 합니다. 인터넷상의 낯선 사람들 사이보다는 팀 내부에서 훨씬 더 효과적입니다.
- 불필요하게 복사 붙여넣기한 지침으로 초반부터 맥락을 과도하게 복잡하게 만드는 경향이 있는데, 제 경험상 점진적으로 구축해 나가는 것이 가장 좋습니다.
- 경험 수준에 따라 필요한 규칙과 지침이 다를 수 있습니다.
- 낯선 사람의 코드를 많이 베껴서 맥락을 제대로 파악하지 못하면, 의도치 않게 지시사항을 반복하거나 기존 내용과 모순되는 말을 할 수도 있고, 단순히 지시사항을 따랐을 뿐인 코딩 에이전트를 쓸모없다고 탓할 수도 있습니다.

# 주의하세요: 통제에 대한 환상

- 이름과는 달리, 궁극적으로 이것은 진정한 의미의 엔지니어링은 아니다. 
- 에이전트가 모든 지침과 안내를 받더라도, 실행 여부는 여전히 LLM이 이를 얼마나 잘 해석하는지에 달려 있다.
- 컨텍스트 엔지니어링은 코딩 에이전트의 효율성을 높이고 유용한 결과를 도출할 확률을 상당히 증가시킬 수 있다. 
- 하지만 때때로 사람들은 이러한 기능을 "X를 수행하도록 보장"하거나 "환상을 방지"하는 등의 표현으로 설명합니다. 
- **그러나 LLM이 관여하는 한, 우리는 어떤 것도 확신 할 수 없으며, 여전히 확률에 기반하여 생각 하고 적절한 수준의 인간 감독을 선택해야 한다.**
