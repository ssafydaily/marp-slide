# Harness

## 에이전트 = 모델 + 하네스

> - 당신이 모델이 아니라면, 당신은 하네스입니다.
> - 모델에는 지능이 담겨 있고, 하네스는 그 지능을 유용하게 만들어주는 시스템

- 하네스는 모델 자체를 제외한 모든 코드, 구성 및 실행 로직을 의미
- 원시 모델은 에이전트가 아니다
- 하네스가 상태, 도구 실행, 피드백 루프 및 강제 가능한 제약 조건과 같은 요소를 제공하면 에이전트가 된다.

# 모델 관점에서 하네스

- 모델이 기본적으로 수행할 수 **없는** 작업을 에이전트가 수행하는 경우 
=> 바로 이 부분에서 하네스가 필요
- 모델은 (대부분) 텍스트, 이미지, 오디오, 비디오와 같은 데이터를 입력받아 텍스트를 출력합

> 기본적으로 모델은 다음 작업을 수행할 수 없다
> - 상호 작용 전반에 걸쳐 안정적인 상태를 유지
> - 코드 실행
> - 실시간 정보에 접근
> - 작업 완료를 위해 환경을 설정하고 패키지를 설치
> 
- 이 모든 것은 하네스 레벨 기능 

-----------
# cont.

- LLM의 구조는 유용한 작업을 수행하도록 모델을 감싸는 메커니즘이 필요
  - 예> **채팅**과 같은 제품 UX를 구현하려면 이전 메시지를 추적하고 새 사용자 메시지를 추가하는 while 루프로 모델 감싸기

- 핵심 아이디어는 원하는 에이전트 동작을 하네스의 실제 기능으로 변환하는 것

--------
# 원하는 에이전트 동작에서 역으로 접근

- 하네스 엔지니어링은 인간이 에이전트의 행동을 안내하는 데 유용한 사전 정보를 주입할 수 있도록 돕는다

- 모델의 능력이 향상됨에 따라, 하네스는 모델을 정밀하게 확장하고 수정하여 이전에는 불가능했던 작업을 가능하게 한다.

### 우리가 원하는 동작(또는 수정하고 싶은 동작) → 디자인을 활용하여 모델이 이를 달성하게 한다.

# 영구 저장 및 컨텍스트 관리 - 파일 시스템

> 에이전트는 뛰어난 저장창치가 필요
> - 에이전트가 실제 데이터와 상호 작용
> - 문맥에 맞지 않는 정보 제거
> - 세션 간 작업을 지속하기


## 파일 시스템 추상화 및 파일 시스템 운영 도구가 포함된 하네스가 필요

- 에이전트는 데이터, 코드 및 문서를 읽을 수 있는 작업 공간을 제공받는다.
- 모든 것을 컨텍스트에 유지하는 대신 작업을 점진적으로 추가하고 분산한다.
- 에이전트는 중간 출력을 저장하고 단일 세션보다 오래 지속되는 상태를 유지
- 여러 에이전트와 사람이 공유 파일을 통해 협업하도록 지원

---------------

# Bash와 코드를 범용 도구로 활용
- 인간이 모든 도구를 미리 설계할 필요 없이 에이전트가 자율적으로 문제를 해결하려면?

- 에이전트 주요 실행 패턴은 [ReAct 루프](https://docs.langchain.com/oss/python/langchain/agents)
  - 모델이 추론, 도구 호출을 통해 작업 수행, 결과 관찰 후 while 루프에서 이 과정을 반복

- 필요한 모든 도구를 사용자가 직접 개발하는 대신, 에이전트에게 **bash**와 같은 범용 도구를 제공하는 것이 더 나은 해결책

=> Bash와 코드 실행을 결합한 방식은 모델에 컴퓨터를 부여 하고 나머지는 자율적으로 해결하도록 하는 데 있어 큰 진전
- 코드 실행은 자율적인 문제 해결을 위한 기본적이고 범용적인 전략

-------------------
# 작업 실행 및 검증 - 샌드박스 및 도구

- 샌드박스는 에이전트에게 안전한 운영 환경을 제공
  - 에이전트는 로컬에서 실행하는 대신 샌드박스에 연결하여 코드를 실행하고, 파일을 검사하고, 종속성을 설치하고, 작업을 완료
- 에이전트가 유용한 작업을 수행할 수 있도록 도구를 구성하는 것은 하네스의 역할
  - 언어 런타임 및 패키지 사전 설치, Git 및 테스트용 CLI, 웹 상호 작용 및 검증을 위한 브라우저 설치

--------------------
# 기억과 지속적인 학습을 위한 탐구

- 모델 가중치를 수정할 수 없는 상황에서 **지식을 추가하는** 유일한 방법은 **컨텍스트 주입**

-  최신 지식을 얻기 위해 웹 검색 및 [Context7](https://context7.com/?ref=blog.langchain.com) 과 같은 MCP 도구 는 에이전트가 지식 제한을 넘어 새로운 라이브러리 버전이나 학습이 중단될 당시에는 존재하지 않았던 최신 데이터에 접근

-------------------
# Context 왜곡과 싸우기
- 에이전트 성능은 작업 과정 동안 저하되어서는 안 됩니다.

- (오늘날) 하네스는 주로 효과적인 컨텍스트 엔지니어링을 위한 전달 메커니즘으로 사용

- **압축(Compaction)**은 컨텍스트 창이 거의 가득 찼을 때 어떻게 처리해야 하는지와 연관

- **툴 호출 오프로딩**은 유용한 정보 없이 컨텍스트 창을 어지럽힐 수 있는 대규모 툴 출력의 
영향을 줄이는 데 도움

- **스킬**은 에이전트 시작 시 컨텍스트에 너무 많은 도구나 **MCP** 서버가 로드되어 에이전트가 작업을 시작하기 전에 성능이 저하되는 문제를 해결
  - 점진적 공개를 통해 이 문제를 해결하는 하네스 수준의 기본 요소

# 하네스의 미래

- 모델 훈련과 하네스 설계의 결합
  - `Claude Code`나 `Codex` 같은 최신 에이전트 제품은 모델과 하네스를 함께 사용하여 사후 학습을 진행
  - 모델은 파일 시스템 작업, bash 실행, 계획 수립, 하위 에이전트를 사용한 병렬 작업 처리 등 모델의 기본적 성능을 향상

# 하네스 엔지니어링의 미래

- 모델의 기능이 향상됨에 따라 현재 시스템에 존재하는 일부 요소들이 모델 자체에 통합될 것
- 모델은 계획 수립, 자체 검증, 장기적인 일관성 유지 기능을 기본적으로 더 잘 수행
  - 예를 들어 컨텍스트 주입이 덜 필요
- 시간이 지남에 따라 하네스의 중요성이 줄어들 것

> [참고]
> - 랭체인(LangChain)에서는 하네스 엔지니어링 연구를 활발히 진행
> - 하네스 구축 라이브러리인 [`deepagents`](https://docs.langchain.com/oss/python/deepagents/overview)를 개선
