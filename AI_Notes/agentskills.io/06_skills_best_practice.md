# 스킬 제작자를 위한 모범 사례

- 작업에 적합하고 잘 조정된 스킬을 작성하는 방법.

## 실제 전문성에서 출발하기

- 스킬 제작시 흔히 빠지는 함정은 도메인 특화 맥락을 제공하지 않고 LLM에게 스킬 생성을 요청하는 것
  — LLM의 일반 학습 지식에만 의존하는 방식입니다. 그 결과는 구체적인 API 패턴, 엣지 케이스, 프로젝트 관례 대신 모호하고 일반적인 절차("오류를 적절히 처리하라", "인증 모범 사례를 따르라")가 됩니다.

- 효과적인 스킬은 실제 전문성에 기반합니다. 핵심은 도메인 특화 맥락을 생성 과정에 투입하는 것입니다.


---------------

### 실제 작업에서 추출하기

에이전트와의 대화에서 실제 작업을 완수하면서 맥락, 수정 사항, 선호도를 제공하세요. 
그런 다음 재사용 가능한 패턴을 스킬로 추출합니다. 

다음 사항에 주목하세요:

- **효과가 있었던 단계** — 성공으로 이어진 일련의 행동
- **수행한 수정** — 에이전트의 접근 방식을 바꾼 지점 (예: "Y 대신 X 라이브러리 사용", "Z 엣지 케이스 확인")
- **입출력 형식** — 데이터가 들어오고 나갈 때의 모습
- **제공한 맥락** — 에이전트가 사전에 알지 못했던 프로젝트 특화 사실, 관례, 또는 제약 사항

----------------

### 기존 프로젝트 산출물에서 합성하기

- 기존 지식 체계가 있다면, LLM에 투입하여 스킬을 합성하도록 요청할 수 있습니다. 
- 팀의 실제 장애 보고서와 런북에서 합성된 데이터 파이프라인 스킬은 일반적인 "데이터 엔지니어링 모범 사례" 문서에서 합성된 것보다 성능이 뛰어납니다. 
- 실제 스키마, 장애 유형, 복구 절차를 담고 있기 때문입니다. 핵심은 일반적인 참조가 아닌 프로젝트 특화 자료입니다.

좋은 원본 자료의 예:

- 내부 문서, 런북, 스타일 가이드
- API 명세, 스키마, 설정 파일
- 코드 리뷰 댓글과 이슈 트래커 (반복되는 우려 사항과 리뷰어 기대치 파악)
- 버전 관리 이력, 특히 패치와 수정 내역 (실제 변경된 내용을 통해 패턴 파악)
- 실제 장애 사례와 해결 방법

-------------------

## 실제 실행으로 정제하기

- 스킬의 첫 번째 초안은 보통 정제가 필요합니다. 
- 실제 작업에 스킬을 실행한 후, 결과물 — 실패뿐만 아니라 전체 결과 — 을 생성 과정에 다시 투입합니다. 

> 다음을 물어보세요: 무엇이 거짓 양성을 유발했나? 무엇이 놓쳤나? 무엇을 줄일 수 있나?

- 단 한 번의 실행-후-수정 으로 품질을 눈에 띄게 향상시키며, 복잡한 도메인은 여러 번의 반복이 필요할 수 있다.

<div class="callout tip">
  <div class="callout-title">
  
  TIP

  </div>
  
  - 최종 출력만이 아닌 에이전트 실행 추적을 읽어보세요.
  - 에이전트가 비생산적인 단계에 시간을 낭비한다면, 일반적인 원인은 다음과 같습니다: 
    - 너무 모호한 지시 (에이전트가 효과적인 방법을 찾기 전에 여러 접근법을 시도)
    - 현재 작업에 적용되지 않는 지시 (에이전트가 그럼에도 따름)
    - 또는 명확한 기본값 없이 제시된 너무 많은 옵션.

- 보다 체계적인 반복 접근법(테스트 케이스, 어서션, 채점 포함)은 [스킬 품질 평가]()를 고려하시오.

-----------------

## 컨텍스트를 현명하게 사용하기

- 스킬이 활성화되면, `SKILL.md`의 전체 내용이 대화 이력, 시스템 컨텍스트, 기타 활성 스킬과 함께 에이전트의 컨텍스트 창에 로드됩니다. 
- 스킬의 모든 토큰은 해당 창의 다른 모든 내용과 에이전트의 주의(attention)를 놓고 경쟁합니다.

### 에이전트가 부족한 것을 추가하고, 아는 것은 생략하기

- 에이전트가 스킬 없이는 알 수 없는 것에 집중하세요: 
  - 프로젝트 특화 관례
  - 도메인 특화 절차
  - 명확하지 않은 엣지 케이스
  - 사용할 특정 도구나 API

---------------

- PDF가 무엇인지, HTTP가 어떻게 작동하는지, 데이터베이스 마이그레이션이 무엇인지 설명할 필요가 없다.

  ````markdown
  <!-- 너무 장황함 — 에이전트는 이미 PDF가 무엇인지 알고 있음 -->
  ## PDF 텍스트 추출
  PDF (Portable Document Format) 파일은 텍스트, 이미지 및 기타 내용을 포함하는
  일반적인 파일 형식입니다. PDF에서 텍스트를 추출하려면 라이브러리를 사용해야
  합니다. pdfplumber는 대부분의 경우를 잘 처리하기 때문에 권장됩니다.

  <!-- 더 나음 — 에이전트가 스스로 알 수 없는 것으로 바로 건너뜀 -->
  ## PDF 텍스트 추출
  텍스트 추출에는 pdfplumber를 사용합니다. 스캔된 문서의 경우,
  pytesseract와 함께 pdf2image로 대체합니다.

  ```python
  import pdfplumber

  with pdfplumber.open("file.pdf") as pdf:
      text = pdf.pages[0].extract_text()
  ```
  ````
-----------------

> **각 내용에 대해 스스로 질문하기:**
>  - "이 지시 없이 에이전트가 이것을 틀리게 할까?" 답이 아니오라면 삭제하세요. 
>    - 확실하지 않다면 테스트하세요. 
>  - 에이전트가 스킬 없이도 이미 전체 작업을 잘 처리한다면, 스킬이 가치를 추가하지 않을 수 있습니다. 
>  - 이를 체계적으로 테스트하는 방법은 [스킬 출력 품질 평가]()를 확인하세요.

-----------------

### 일관된 단위 설계하기

- 스킬이 커버해야 할 범위를 결정하는 것은 함수가 무엇을 해야 하는지 결정하는 것과 같다: 
  - 다른 스킬과 잘 조합되는 일관된 작업 단위를 캡슐화하길 원합니다. 
  - 너무 좁게 범위를 정한 스킬은 단일 작업에 여러 스킬을 로드하게 만들어 오버헤드와 충돌하는 지시 위험을 초래합니다. 
  - 너무 넓게 범위를 정한 스킬은 정확하게 활성화하기 어렵습니다. 
    - 데이터베이스 쿼리와 결과 형식화를 위한 스킬은 하나의 일관된 단위일 수 있지만, 
    - 데이터베이스 관리까지 커버하는 스킬은 아마도 너무 많은 것을 하려는 것입니다.

### 적당한 수준의 세부 사항 목표로 하기

- 지나치게 포괄적인 스킬은 도움이 되기보다 해가 될 수 있습니다 
  - 에이전트는 관련 내용을 추출하는 데 어려움을 겪고 현재 작업에 적용되지 않는 지시에 의해 촉발된 비생산적인 경로를 추구할 수 있습니다. 
  - 작동 예시가 있는 간결하고 단계별 지침이 포괄적인 문서보다 성능이 뛰어난 경향이 있습니다. 
  - 모든 엣지 케이스를 커버하게 되면, 대부분이 에이전트 자체 판단으로 더 잘 처리될 수 있는지 고려해 보세요.

-----------------

### 점진적 공개로 대형 스킬 구조화하기

- **명세서** 는 `SKILL.md`를 500줄, 5,000 토큰 미만으로 유지하도록 권장
— 모든 실행에서 에이전트에게 필요한 핵심 지시만 포함합니다. 
— 스킬이 합당하게 더 많은 내용을 필요로 할 때, 상세 참조 자료를 `references/` 또는 유사한 디렉토리의 별도 파일로 이동하세요.

- 핵심은 에이전트에게 *언제* 각 파일을 로드해야 하는지 알려주는 것입니다. 
  - "API가 200이 아닌 상태 코드를 반환하면 `references/api-errors.md`를 읽어라"가 "자세한 내용은 references/를 참조하라"보다 더 유용

 >이를 통해 에이전트는 선행 로드 대신 필요에 따라 컨텍스트를 로드할 수 있으며, 이것이 [**점진적 공개(progressive disclosure)**](04_agent_skills.md) 의 설계 방식

--------------------

## 제어 보정하기

- 스킬의 모든 부분이 동일한 수준의 규범성을 필요로 하지는 않습니다. 
- 작업의 취약성에 맞게 지시의 구체성을 맞추세요.

### 취약성에 맞게 구체성 조정하기

- **에이전트에게 자유를 주세요** : 여러 접근법이 유효하고 작업이 변형을 허용할 때. 
  - 유연한 지시의 경우, 엄격한 지시보다 *왜*를 설명하는 것이 더 효과적일 수 있습니다 
  — 지시 뒤의 목적을 이해하는 에이전트는 더 나은 맥락 의존적 결정을 합니다. 
  — 코드 리뷰 스킬은 정확한 단계를 규정하지 않고도 무엇을 찾아볼지 설명할 수 있습니다:

  ```markdown
  ## 코드 리뷰 프로세스

  1. 모든 데이터베이스 쿼리에서 SQL 인젝션 확인 (매개변수화된 쿼리 사용)
  2. 모든 엔드포인트에서 인증 확인 검증
  3. 동시 코드 경로에서 경쟁 조건 확인
  4. 오류 메시지가 내부 세부 정보를 유출하지 않는지 확인
  ```

------------

- **규범적으로** 운영이 취약하거나, 일관성이 중요하거나, 특정 순서를 따라야 할 때:

  ````markdown 
  ## 데이터베이스 마이그레이션

  정확히 이 순서대로 실행:

  ```bash
  python scripts/migrate.py --verify --backup
  ```

  명령을 수정하거나 추가 플래그를 더하지 마세요.
  ````

- 대부분의 스킬은 혼합되어 있습니다. 각 부분을 독립적으로 보정하세요.

----------------

### 메뉴가 아닌 기본값 제공하기

- 여러 도구나 접근법이 가능할 때, 기본값을 선택하고 대안은 간략하게 언급하세요 
— 동등한 옵션으로 제시하지 마세요.

````markdown theme={null}
<!-- 너무 많은 옵션 -->
pypdf, pdfplumber, PyMuPDF, 또는 pdf2image를 사용할 수 있습니다...

<!-- 탈출구가 있는 명확한 기본값 -->
텍스트 추출에 pdfplumber를 사용하세요:

```python
import pdfplumber
```

OCR이 필요한 스캔된 PDF는 대신 pdf2image와 pytesseract를 사용하세요.
````

---------------

### 선언보다 절차 선호하기

스킬은 특정 인스턴스에 대해 ***무엇을 생성할지*** 가 아닌, 에이전트가 문제 유형에 ***어떻게 접근할지*** 를 가르쳐야 합니다. 비교해 보세요:

```markdown 
<!-- 특정 답변 — 이 정확한 작업에만 유용 -->

`orders` 테이블을 `customer_id`로 `customers`에 조인하고,
`region = 'EMEA'`로 필터링하여 `amount` 열을 합산하세요.

<!-- 재사용 가능한 방법 — 모든 분석 쿼리에 적용 -->

1. `references/schema.yaml`에서 스키마를 읽어 관련 테이블 찾기
2. `_id` 외래 키 관례를 사용하여 테이블 조인
3. 사용자 요청의 필터를 WHERE 절로 적용
4. 필요에 따라 숫자 열을 집계하고 마크다운 표로 형식화
```

## 효과적인 지시를 위한 패턴

이것은 스킬 내용을 구조화하기 위한 재사용 가능한 기법입니다. 모든 스킬이 이것들을 전부 필요로 하지는 않습니다 — 작업에 맞는 것을 사용하세요.

### 주의사항 섹션

- 많은 스킬에서 가장 높은 가치를 가진 내용은 **주의사항 목록** 입니다 
— 합리적인 가정을 벗어나는 환경 특화 사실들. 
— 이것들은 일반적인 조언("오류를 적절히 처리하라")이 아니라, 그 외에는 에이전트가 만들 실수에 대한 구체적인 수정입니다:

-----------------

```markdown theme={null}
## 주의사항

- `users` 테이블은 소프트 삭제를 사용합니다. 
  - 쿼리에는 `WHERE deleted_at IS NULL`을 포함해야 하며, 
  - 그렇지 않으면 비활성화된 계정이 결과에 포함됩니다.

- 사용자 ID는 데이터베이스에서 `user_id`, 인증 서비스에서 `uid`,
  결제 API에서 `accountId`입니다. 세 가지 모두 동일한 값을 가리킵니다.

- `/health` 엔드포인트는 데이터베이스 연결이 끊어져도 웹 서버가
  실행 중이면 200을 반환합니다. 전체 서비스 상태를 확인하려면 `/ready`를 사용하세요.
```

---------------

- 에이전트가 상황을 만나기 전에 읽을 수 있도록 주의사항을 `SKILL.md`에 유지하세요. 
- 별도 참조 파일도 에이전트에게 언제 로드해야 하는지 알려준다면 효과적이나, 
- 불병확한 문제의 경우 에이전트가 트리거를 인식하지 못할 수 있다.

<div class="callout tip">
  <div class="callout-title">
  
  TIP

  </div>
  
  - 에이전트가 수정해야 하는 실수를 저지를 때, 수정 사항을 주의사항 섹션에 추가하세요. 
  - 이것이 스킬을 반복적으로 개선하는 가장 직접적인 방법 중 하나입니다.

</div>

### 출력 형식 템플릿

- 에이전트가 특정 형식으로 출력을 생성해야 할 때, 템플릿을 제공하세요. 
  - 이것이 형식을 산문으로 설명하는 것보다 더 신뢰할 수 있습니다. 
  - 에이전트가 구체적인 구조에 대해 패턴 매칭을 잘 하기 때문입니다. 

- 짧은 템플릿은 `SKILL.md`에 인라인으로 넣을 수 있습니다. 
- 긴 템플릿이나 특정 경우에만 필요한 템플릿은 `assets/`에 저장하고 `SKILL.md`에서 참조하여 필요할 때만 로드되도록 하세요.

---------------

- **보고서 출력 형식 예시**
````markdown
## 보고서 구조

특정 분석에 맞게 섹션을 조정하면서 이 템플릿을 사용하세요:

```markdown
# [분석 제목]
````

### 다단계 워크플로우를 위한 체크리스트

- **명시적 체크리스트** 는 에이전트가 **진행 상황을 추적** 하고 단계를 건너뛰지 않도록 도와줍니다. 
- 특히 단계에 종속성이나 검증 관문이 있을 때 효과적입니다.

```markdown theme={null}
## 양식 처리 워크플로우

진행 상황:
- [ ] 1단계: 양식 분석 (`scripts/analyze_form.py` 실행)
- [ ] 2단계: 필드 매핑 생성 (`fields.json` 편집)
- [ ] 3단계: 매핑 검증 (`scripts/validate_fields.py` 실행)
- [ ] 4단계: 양식 작성 (`scripts/fill_form.py` 실행)
- [ ] 5단계: 출력 확인 (`scripts/verify_output.py` 실행)
```

-----------------

### 검증 루프

- 에이전트에게 진행하기 전에 자체 작업을 검증하도록 지시하세요. 
- 패턴은: 작업 수행, 검증기 실행 (스크립트, 참조 체크리스트, 또는 자체 검사), 문제 수정, 검증이 통과될 때까지 반복입니다.

```markdown theme={null}
## 편집 워크플로우

1. 편집 수행
2. 검증 실행: `python scripts/validate.py output/`
3. 검증 실패 시:
   - 오류 메시지 검토
   - 문제 수정
   - 검증 다시 실행
4. 검증이 통과될 때만 진행
```

- 참조 문서도 "검증기" 역할을 할 수 있습니다 — 에이전트에게 마무리하기 전에 참조에 대해 작업을 확인하도록 지시하세요.

-----------------

### 계획-검증-실행

- 에이전트가 구조화된 형식으로 중간 계획을 작성하고, 진실의 원천에 대해 검증한 후에만 실행하도록 하세요.

  ```markdown 
  ## PDF 양식 작성

  1. 양식 필드 추출: `python scripts/analyze_form.py input.pdf` → `form_fields.json`
    (모든 필드 이름, 유형, 필수 여부 목록)
  2. 각 필드 이름을 의도한 값에 매핑하는 `field_values.json` 생성
  3. 검증: `python scripts/validate_fields.py form_fields.json field_values.json`
    (모든 필드 이름이 양식에 존재하는지, 유형이 호환되는지, 필수 필드가
    누락되지 않았는지 확인)
  4. 검증 실패 시, `field_values.json`을 수정하고 재검증
  5. 양식 작성: `python scripts/fill_form.py input.pdf field_values.json output.pdf`
  ```

핵심 재료는 3단계입니다: 계획(`field_values.json`)을 진실의 원천(`form_fields.json`)에 대해 확인하는 검증 스크립트. "필드 'signature\_date'를 찾을 수 없음 — 사용 가능한 필드: customer\_name, order\_total, signature\_date\_signed"와 같은 오류는 에이전트가 자체 수정할 충분한 정보를 제공합니다.

### 재사용 가능한 스크립트 번들링

- [스킬을 반복할 때] 테스트 케이스 전반에 걸쳐 에이전트의 실행 추적을 비교하세요.
- 에이전트가 매 실행마다 동일한 로직을 독립적으로 재발명하는 것을 알아차린다면 
— 차트 작성, 특정 형식 파싱, 출력 검증 — 이것은 한 번 테스트된 스크립트를 작성하고 `scripts/`에 번들링해야 한다는 신호입니다.

스크립트 설계 및 번들링에 대한 자세한 내용은 [스킬에서 스크립트 사용하기](/skill-creation/using-scripts)를 참조하세요.
