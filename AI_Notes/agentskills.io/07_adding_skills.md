# 에이전트에 스킬 지원 추가하는 방법

> AI 에이전트 또는 개발 도구에 Agent Skills 지원을 추가하는 가이드.

- 이 가이드는 AI 에이전트 또는 개발 도구에 Agent Skills 지원을 추가하는 전체 과정을 다룹니다. 
- 스킬 탐색, 모델에 스킬 알리기, 컨텍스트에 콘텐츠 로드, 그리고 해당 콘텐츠를 지속적으로 유효하게 유지하는 방법까지 포함합니다.

> 에이전트 아키텍처에 관계없이 핵심 통합 방식은 동일하며, 세부 사항은 두 가지 요소에 따라 달라진다:

- **스킬이 어디에 위치하는가?** 
  - 로컬에서 실행되는 에이전트는 파일시스템에서 스킬 디렉토리를 스캔할 수 있다. 
  - 클라우드 호스팅 또는 샌드박스 에이전트는 API, 원격 레지스트리, 또는 번들 자산과 같은 대안적인 탐색 메커니즘이 필요
- **모델이 스킬 콘텐츠에 어떻게 접근하는가?** 
  - 모델에 파일 읽기 기능이 있다면 `SKILL.md` 파일을 직접 읽을 수 있다. 
  - 그렇지 않은 경우는 전용 도구를 제공하거나 스킬 콘텐츠를 프로그래밍 방식으로 프롬프트에 주입한다.

-------------------------

- **사전 조건**: `SKILL.md` 파일 형식, 프론트매터 필드, 디렉토리 규약을 정의하는 **Agent Skills 명세** 에 대한 이해가 필요

## 핵심 원칙: 점진적 공개 (Progressive Disclosure)

- 스킬 호환 에이전트는 모두 동일한 3단계 로딩 전략을 따릅니다:

  | 단계           | 로드되는 내용              | 시점                        | 토큰 비용                    |
  | -------------- | -------------------------- | --------------------------- | ---------------------------- |
  | 1. 카탈로그    | 이름 + 설명                | 세션 시작 시                | 스킬당 약 50-100 토큰        |
  | 2. 지침        | 전체 `SKILL.md` 본문       | 스킬 활성화 시              | 5000 토큰 미만 (권장)        |
  | 3. 리소스      | 스크립트, 참조 자료, 자산  | 지침에서 참조될 때          | 가변                         |

- 모델은 처음부터 카탈로그를 보기 때문에 어떤 스킬을 사용할 수 있는지 알 수 있다. 
- 스킬이 관련 있다고 판단하면 전체 지침을 로드한다. 지침에서 지원 파일을 참조하면 필요에 따라 개별적으로 로드합니다.

이를 통해 기본 컨텍스트를 작게 유지하면서도 필요할 때 전문 지식에 접근할 수 있습니다. 20개의 스킬이 설치된 에이전트는 주어진 대화에서 실제로 사용되는 스킬만 토큰 비용을 지불합니다.

--------------------------

## 1단계: 스킬 탐색

- 세션 시작 시 모든 사용 가능한 스킬을 찾고 메타데이터를 로드합니다.

### 스캔 위치

- 스캔할 디렉토리는 에이전트의 환경에 따라 다릅니다. 로컬에서 실행되는 대부분의 에이전트는 최소 두 가지 범위를 스캔합니다:

- **프로젝트 수준** (작업 디렉토리 기준): 프로젝트 또는 저장소에 특화된 스킬
- **사용자 수준** (홈 디렉토리 기준): 특정 사용자의 모든 프로젝트에서 사용 가능한 스킬

> 관리자가 배포한 조직 전체 스킬이나 에이전트 자체에 번들된 스킬 등 다른 범위도 가능하다. 적절한 범위는 에이전트의 배포 모델에 따라 다르다.

------------------------

- 각 범위 내에서 **클라이언트 전용 디렉토리**와 **`.agents/skills/` 규약** 모두를 스캔하는 것을 고려하세요:

| 범위      | 경로                                  | 목적                          |
| --------- | ------------------------------------- | ----------------------------- |
| 프로젝트  | `<project>/.<your-client>/skills/`    | 클라이언트 고유 위치          |
| 프로젝트  | `<project>/.agents/skills/`           | 크로스 클라이언트 상호운용성  |
| 사용자    | `~/.<your-client>/skills/`            | 클라이언트 고유 위치          |
| 사용자    | `~/.agents/skills/`                   | 크로스 클라이언트 상호운용성  |

- `.agents/skills/` 경로는 크로스 클라이언트 스킬 공유를 위한 규약으로 사용됨. 
- Agent Skills 명세는 스킬 디렉토리의 위치를 강제하지 않지만(내부 내용만 정의), 
-`.agents/skills/`를 스캔하면 다른 호환 클라이언트가 설치한 스킬이 자동으로 보이고, 그 반대도 마찬가지입니다.

<div class="callout tip">
  <div class="callout-title">
  TIP
  </div>  

  - 일부는 실용적인 호환성을 위해 `.claude/skills/`(프로젝트 수준 및 사용자 수준 모두)도 스캔
  - 추가적인 위치로는 git 루트까지의 상위 디렉토리(모노레포에 유용), [XDG] 설정 디렉토리, 사용자 설정 경로 등이 있습니다.

</div>

------------------

### 스캔 대상

- 각 스킬 디렉토리 내에서 **`SKILL.md` 파일을 포함하는 하위 디렉토리**를 찾습니다:

  ```
  ~/.agents/skills/
  ├── pdf-processing/
  │   ├── SKILL.md          ← 탐색됨
  │   └── scripts/
  │       └── extract.py
  ├── data-analysis/
  │   └── SKILL.md          ← 탐색됨
  └── README.md             ← 무시됨 (스킬 디렉토리가 아님)
  ```

> 실용적인 스캔 규칙:

- `.git/`과 `node_modules/` 같은 스킬을 포함하지 않을 디렉토리는 skip
- `.gitignore`를 선택적으로 준수하여 빌드 결과물 스캔을 방지
- 큰 디렉토리 트리에서 무한 스캔을 방지하기 위해 합리적인 제한을 설정 (예: 최대 깊이 4-6단계, 최대 2000개 디렉토리)

--------------------

### 이름 충돌 처리

- 두 스킬이 같은 `name`을 가질 경우 결정적인 우선순위 규칙을 적용

- 기존 구현체 전반에서 통용되는 규약: **프로젝트 수준 스킬이 사용자 수준 스킬보다 우선합니다.**

- 같은 범위 내에서 충돌이 발생하는 경우(예: `<project>/.agents/skills/`와 `<project>/.<your-client>/skills/` 양쪽에서 `code-review`라는 스킬이 발견될 때), 
- 첫 번째 발견 또는 마지막 발견 중 하나를 선택해 일관되게 적용하세요. 
- 스킬이 숨겨졌음을 사용자가 알 수 있도록 충돌 발생 시 경고를 기록하세요.

-----------------

### 신뢰 고려사항

- 프로젝트 수준 스킬은 작업 중인 저장소에서 가져오는데, 이 저장소가 신뢰할 수 없는 것일 수도 있습니다(예: 새로 클론한 오픈소스 프로젝트). 
- 신뢰 확인을 통해 프로젝트 수준 스킬 로딩을 제한하는 것을 고려하세요 
— 사용자가 프로젝트 폴더를 신뢰할 수 있는 것으로 표시한 경우에만 로드합니다. 
— 이렇게 하면 신뢰할 수 없는 저장소가 에이전트의 컨텍스트에 지침을 몰래 주입하는 것을 방지할 수 있습니다.

### 클라우드 호스팅 및 샌드박스 에이전트

- 에이전트가 컨테이너나 원격 서버에서 실행되는 경우, 사용자의 로컬 파일시스템에 접근할 수 없다. 
- 탐색은 스킬 범위에 따라 다르게 작동해야 합니다:

  - **프로젝트 수준 스킬**은 가장 쉬운 경우. 
    - 에이전트가 클론된 저장소에서 작업하는 경우(샌드박스 내부라도), 프로젝트 수준 스킬은 코드와 함께 이동하며 저장소의 디렉토리 트리에서 스캔할 수 있습니다.

----------------------

  - **사용자 수준 및 조직 수준 스킬**은 샌드박스에 존재하지 않는다. 
    - 외부 소스에서 제공해야 합니다 
    — 예를 들어 설정 저장소 클론, 에이전트 설정을 통한 스킬 URL 또는 패키지 수락, 또는 웹 UI를 통한 스킬 디렉토리 업로드 등이 있습니다.

  - **내장 스킬**은 에이전트의 배포 아티팩트 내에 정적 자산으로 패키징하여 외부 접근 없이 모든 세션에서 사용 가능하게 할 수 있습니다.


------------------

## 2단계: `SKILL.md` 파일 파싱

- 탐색된 각 `SKILL.md`에서 메타데이터와 본문 콘텐츠를 추출합니다.

### 프론트매터 추출

`SKILL.md` 파일은 두 부분으로 구성됩니다: `---` 구분자 사이의 YAML 프론트매터, 그리고 닫는 구분자 이후의 마크다운 본문입니다. 파싱 방법:

1. 파일 시작 부분의 `---`와 그 이후의 닫는 `---`를 찾습니다.
2. 사이의 YAML 블록을 파싱합니다. `name`과 `description`(필수), 그리고 선택적 필드를 추출합니다.
3. 닫는 `---` 이후의 모든 내용(앞뒤 공백 제거)이 스킬의 본문 콘텐츠입니다.

프론트매터 필드 전체 목록과 제약 조건은 [명세](/specification)를 참조하세요.

------------

### 잘못된 YAML 처리

다른 클라이언트용으로 작성된 스킬 파일에는 해당 파서가 허용하는 기술적으로 유효하지 않은 YAML이 포함될 수 있습니다. 가장 일반적인 문제는 콜론이 포함된 따옴표 없는 값입니다:

```yaml theme={null}
# 기술적으로 유효하지 않은 YAML — 콜론이 파싱을 방해함
description: 다음 경우에 이 스킬을 사용하세요: 사용자가 PDF에 대해 질문할 때
```

이러한 값을 따옴표로 감싸거나 YAML 블록 스칼라로 변환한 후 재시도하는 폴백을 고려하세요. 이렇게 하면 최소한의 비용으로 크로스 클라이언트 호환성이 향상됩니다.

---------------


### 관대한 유효성 검사

가능한 경우 경고를 기록하되 스킬을 로드하세요:

* 이름이 상위 디렉토리 이름과 일치하지 않음 → 경고, 그래도 로드
* 이름이 64자를 초과함 → 경고, 그래도 로드
* 설명이 없거나 비어 있음 → 스킬 건너뜀(설명은 공개에 필수), 오류 기록
* YAML이 완전히 파싱 불가능 → 스킬 건너뜀, 오류 기록

진단 정보를 기록하여 사용자에게 표면화(디버그 명령, 로그 파일, 또는 UI)할 수 있도록 하되, 사소한 문제로 스킬 로딩을 차단하지 마세요.

<div class="callout tip">
  <div class="callout-title">
  
  TIP
  
  </div>
  
  - **명세** 는 `name` 필드에 대한 엄격한 제약(상위 디렉토리와 일치, 문자 집합, 최대 길이)을 정의합니다. 
  - 위의 관대한 접근 방식은 의도적으로 이를 완화하여 다른 클라이언트용으로 작성된 스킬과의 호환성을 향상시킵니다.

</div>

-------------------------------

### 저장 내용

- 최소한 각 스킬 레코드에는 세 가지 필드가 필요합니다:

| 필드          | 설명                            |
| ------------- | ------------------------------- |
| `name`        | 프론트매터에서 가져옴           |
| `description` | 프론트매터에서 가져옴           |
| `location`    | `SKILL.md` 파일의 절대 경로     |

- 활성화 중 빠른 조회를 위해 `name`을 키로 하는 인메모리 맵에 저장하세요.

탐색 시 **본문**(프론트매터 이후의 마크다운 콘텐츠)도 저장하거나, 활성화 시 `location`에서 읽을 수 있습니다. 저장하면 활성화가 빨라지고, 활성화 시 읽으면 전체적으로 메모리 사용량이 줄고 활성화 사이의 스킬 파일 변경 사항이 반영됩니다.

스킬의 **기본 디렉토리**(위치의 상위 디렉토리)는 나중에 상대 경로를 해석하고 번들된 리소스를 열거하는 데 필요합니다 


--------------------

## 3단계: 모델에 사용 가능한 스킬 공개

- 전체 콘텐츠를 로드하지 않고 어떤 스킬이 있는지 모델에 알립니다. 이것이 [점진적 공개의 1단계]입니다.

##### 스킬 카탈로그 구성

탐색된 각 스킬에 대해 `name`, `description`, 그리고 선택적으로 `location`(`SKILL.md` 파일 경로)을 포함시킵니다. XML, JSON, 또는 글머리 목록 등 스택에 맞는 구조화된 형식을 사용하세요:

  ```xml theme={null}
  <available_skills>
    <skill>
      <name>pdf-processing</name>
      <description>PDF 텍스트 추출, 양식 작성, 파일 병합. PDF 처리 시 사용.</description>
      <location>/home/user/.agents/skills/pdf-processing/SKILL.md</location>
    </skill>
    <skill>
      <name>data-analysis</name>
      <description>데이터셋 분석, 차트 생성, 요약 보고서 작성.</description>
      <location>/home/user/project/.agents/skills/data-analysis/SKILL.md</location>
    </skill>
  </available_skills>
  ```

-----------------

`location` 필드는 두 가지 목적을 제공합니다: 파일 읽기 활성화 지원(4단계 참조)과 스킬 본문의 상대 참조(예: `scripts/evaluate.py`)를 해석하기 위한 기본 경로 제공. 전용 활성화 도구가 결과에 스킬 디렉토리 경로를 제공하는 경우(4단계의 [구조화된 래핑](#구조화된-래핑) 참조), 카탈로그에서 `location`을 생략할 수 있습니다.

각 스킬은 카탈로그에 대략 50-100 토큰을 추가합니다. 수십 개의 스킬이 설치되어 있어도 카탈로그는 작게 유지됩니다.

### 카탈로그 배치 위치

두 가지 일반적인 접근 방식이 있습니다:

**시스템 프롬프트 섹션**: 스킬 사용 방법에 대한 간단한 지침과 함께 카탈로그를 시스템 프롬프트의 레이블이 붙은 섹션으로 추가합니다. 이것이 가장 간단한 방법이며 파일 읽기 도구를 가진 모든 모델에서 작동합니다.

**도구 설명**: 카탈로그를 전용 스킬 활성화 도구의 설명에 포함시킵니다(4단계 참조). 이렇게 하면 시스템 프롬프트가 깔끔해지고 탐색과 활성화가 자연스럽게 연결됩니다.

두 방법 모두 작동합니다. 시스템 프롬프트 배치가 더 간단하고 광범위하게 호환되며, 전용 활성화 도구가 있을 때는 도구 설명 임베딩이 더 깔끔합니다.

### 동작 지침

카탈로그와 함께 모델에게 스킬을 어떻게, 언제 사용하는지 알려주는 짧은 지침 블록을 포함하세요. 문구는 지원하는 활성화 메커니즘에 따라 다릅니다(4단계 참조):

**모델이 파일 읽기로 스킬을 활성화하는 경우:**

```
다음 스킬은 특정 작업에 대한 전문 지침을 제공합니다.
작업이 스킬 설명과 일치하면, 진행하기 전에 파일 읽기 도구를 사용하여
나열된 위치의 SKILL.md를 로드하세요.
스킬이 상대 경로를 참조하는 경우, 스킬 디렉토리(SKILL.md의 상위 폴더)를
기준으로 해석하고 도구 호출에 절대 경로를 사용하세요.
```

**모델이 전용 도구로 스킬을 활성화하는 경우:**

```
다음 스킬은 특정 작업에 대한 전문 지침을 제공합니다.
작업이 스킬 설명과 일치하면, 스킬 이름과 함께 activate_skill 도구를 호출하여
전체 지침을 로드하세요.
```

이 지침은 간결하게 유지하세요. 목표는 스킬이 있다는 것과 로드 방법을 모델에게 알리는 것이며, 로드된 스킬 콘텐츠 자체가 세부 지침을 제공합니다.

### 필터링

일부 스킬은 카탈로그에서 제외해야 합니다. 일반적인 이유:

* 사용자가 설정에서 스킬을 비활성화함
* 권한 시스템이 스킬 접근을 거부함
* 스킬이 모델 기반 활성화를 거부함(예: `disable-model-invocation` 플래그)

필터링된 스킬은 카탈로그에서 **완전히 숨기세요**. 나열하고 활성화 시 차단하는 것보다 낫습니다. 이렇게 하면 모델이 사용할 수 없는 스킬을 로드하려는 시도를 낭비하지 않습니다.

### 사용 가능한 스킬이 없는 경우

스킬이 탐색되지 않으면 카탈로그와 동작 지침을 완전히 생략하세요. 빈 `<available_skills/>` 블록을 표시하거나 유효한 옵션 없이 스킬 도구를 등록하지 마세요 — 이는 모델을 혼란스럽게 합니다.

## 4단계: 스킬 활성화

모델 또는 사용자가 스킬을 선택하면 대화 컨텍스트에 전체 지침을 제공합니다. 이것이 [점진적 공개의 2단계](#핵심-원칙-점진적-공개-progressive-disclosure)입니다.

### 모델 기반 활성화

대부분의 구현체는 모델 자체의 판단을 활성화 메커니즘으로 사용하며, 하네스 측 트리거 매칭이나 키워드 감지를 구현하지 않습니다. 모델은 카탈로그를 읽고(3단계에서), 스킬이 현재 작업과 관련 있다고 판단하면 로드합니다.

두 가지 구현 패턴이 있습니다:

**파일 읽기 활성화**: 모델이 카탈로그의 `SKILL.md` 경로로 표준 파일 읽기 도구를 호출합니다. 특별한 인프라가 필요 없으며, 에이전트의 기존 파일 읽기 기능으로 충분합니다. 모델은 도구 결과로 파일 콘텐츠를 받습니다. 모델이 파일에 접근할 수 있을 때 가장 간단한 방법입니다.

**전용 도구 활성화**: 스킬 이름을 받아 콘텐츠를 반환하는 도구(예: `activate_skill`)를 등록합니다. 모델이 직접 파일을 읽을 수 없을 때 필수이며, 읽을 수 있을 때도 선택적으로 유용합니다. 파일 직접 읽기 대비 장점:

* 반환 콘텐츠 제어 — 예: YAML 프론트매터 제거 또는 유지(아래 [모델이 받는 내용](#모델이-받는-내용) 참조)
* 컨텍스트 관리 중 식별을 위한 구조화된 태그로 콘텐츠 래핑
* 번들된 리소스(예: `references/*`) 지침과 함께 나열
* 권한 적용 또는 사용자 동의 요청
* 분석을 위한 활성화 추적

<Tip>
  전용 활성화 도구를 사용하는 경우, `name` 매개변수를 유효한 스킬 이름 집합으로 제한하세요(예: 도구 스키마의 enum으로). 이렇게 하면 모델이 존재하지 않는 스킬 이름을 환각하는 것을 방지합니다. 사용 가능한 스킬이 없으면 도구를 등록하지 마세요.
</Tip>

### 사용자 명시적 활성화

사용자는 모델이 판단하기를 기다리지 않고 직접 스킬을 활성화할 수 있어야 합니다. 가장 일반적인 패턴은 하네스가 가로채는 **슬래시 명령 또는 멘션 구문**(`/skill-name` 또는 `$skill-name`)입니다. 구체적인 구문은 자유롭게 선택하면 됩니다 — 핵심은 하네스가 조회와 주입을 처리하므로 모델이 활성화 작업을 수행하지 않아도 스킬 콘텐츠를 받는다는 것입니다.

자동 완성 위젯(사용자가 입력하는 동안 사용 가능한 스킬 나열)도 이를 탐색 가능하게 만들 수 있습니다.

### 모델이 받는 내용

스킬이 활성화되면 모델은 스킬의 지침을 받습니다. 콘텐츠의 구체적인 형태에는 두 가지 옵션이 있습니다:

**전체 파일**: 모델이 YAML 프론트매터를 포함한 전체 `SKILL.md`를 봅니다. 이는 파일 읽기 활성화의 자연스러운 결과이며, 전용 도구에서도 유효한 선택입니다. 프론트매터에는 활성화 시 유용한 필드가 포함될 수 있습니다 — 예를 들어 [`compatibility`](/specification#compatibility-field) 필드는 스킬 지침 실행 방법에 영향을 줄 수 있는 환경 요구사항을 기록합니다.

**본문만(프론트매터 제거)**: 하네스가 YAML 프론트매터를 파싱하고 제거하여 마크다운 지침만 반환합니다. 전용 활성화 도구를 사용하는 기존 구현체 대부분이 이 방식을 사용합니다 — 탐색 중에 `name`과 `description`을 추출한 후 프론트매터를 제거합니다.

두 방식 모두 실제로 작동합니다.

### 구조화된 래핑

전용 활성화 도구를 사용하는 경우 스킬 콘텐츠를 식별 태그로 래핑하는 것을 고려하세요. 예시:

```xml theme={null}
<skill_content name="pdf-processing">
# PDF 처리

## 이 스킬을 사용하는 경우
사용자가 PDF 파일을 다뤄야 할 때 이 스킬을 사용하세요...

[SKILL.md 본문의 나머지 내용]

스킬 디렉토리: /home/user/.agents/skills/pdf-processing
이 스킬의 상대 경로는 스킬 디렉토리를 기준으로 합니다.

<skill_resources>
  <file>scripts/extract.py</file>
  <file>scripts/merge.py</file>
  <file>references/pdf-spec-summary.md</file>
</skill_resources>
</skill_content>
```

이것의 실용적인 이점:

* 모델이 스킬 지침을 다른 대화 콘텐츠와 명확히 구분할 수 있음
* 하네스가 컨텍스트 압축 중 스킬 콘텐츠를 식별할 수 있음(5단계)
* 번들된 리소스가 즉시 로드되지 않고도 모델에 노출됨

### 번들된 리소스 나열

전용 활성화 도구가 스킬 콘텐츠를 반환할 때 스킬 디렉토리의 지원 파일(스크립트, 참조 자료, 자산)도 열거할 수 있지만 **즉시 읽어서는 안 됩니다**. 모델은 스킬 지침에서 참조할 때 파일 읽기 도구를 사용하여 특정 파일을 필요에 따라 로드합니다.

대형 스킬 디렉토리의 경우 목록에 제한을 두고 불완전할 수 있음을 표시하는 것을 고려하세요.

### 권한 허용 목록

에이전트에 파일 접근을 제한하는 권한 시스템이 있는 경우, **스킬 디렉토리를 허용 목록에 추가**하여 모델이 번들된 리소스를 읽을 때 사용자 확인 요청이 발생하지 않게 하세요. 이렇게 하지 않으면 `SKILL.md` 외의 리소스가 포함된 스킬을 사용할 때마다 권한 대화상자가 나타나 흐름이 끊어집니다.

## 5단계: 시간 경과에 따른 스킬 컨텍스트 관리

스킬 지침이 대화 컨텍스트에 들어오면, 세션 동안 유효하게 유지하세요.

### 컨텍스트 압축으로부터 스킬 콘텐츠 보호

에이전트가 컨텍스트 창이 채워질 때 오래된 메시지를 잘라내거나 요약하는 경우, **스킬 콘텐츠를 제거 대상에서 제외**하세요. 스킬 지침은 지속적인 동작 지침이며, 대화 중간에 잃어버리면 에이전트의 성능이 눈에 띄는 오류 없이 조용히 저하됩니다. 모델은 계속 작동하지만 스킬이 제공한 전문 지침 없이 동작합니다.

일반적인 접근 방식:

* 스킬 도구 출력을 보호됨으로 표시하여 제거 알고리즘이 건너뛰도록 함
* 4단계의 [구조화된 태그](#구조화된-래핑)를 사용하여 스킬 콘텐츠를 식별하고 압축 중 보존

### 중복 활성화 방지

현재 세션에서 어떤 스킬이 활성화되었는지 추적하는 것을 고려하세요. 모델(또는 사용자)이 이미 컨텍스트에 있는 스킬을 로드하려고 하면 재주입을 건너뛰어 같은 지침이 대화에 여러 번 나타나는 것을 방지할 수 있습니다.

### 하위 에이전트 위임 (선택적)

이것은 일부 클라이언트만 지원하는 고급 패턴입니다. 스킬 지침을 메인 대화에 주입하는 대신, 스킬이 **별도의 하위 에이전트 세션**에서 실행됩니다. 하위 에이전트는 스킬 지침을 받아 작업을 수행하고, 작업 요약을 메인 대화에 반환합니다.

이 패턴은 스킬의 워크플로우가 전용 집중 세션의 이점을 받을 만큼 복잡한 경우에 유용합니다.
