# 스킬 출력 품질 평가하기

> 평가 기반 반복을 사용하여 스킬이 좋은 출력을 생성하는지 테스트하는 방법.

스킬을 작성하고, 프롬프트에 시도해보고, 작동하는 것처럼 보였습니다. 하지만 다양한 프롬프트에서, 엣지 케이스에서, 스킬이 없는 것보다 더 나은 경우에 안정적으로 작동하나요? 구조화된 평가(evals)를 실행하면 이러한 질문에 답할 수 있고 스킬을 체계적으로 개선할 피드백 루프를 제공합니다.

## 테스트 케이스 설계하기

테스트 케이스는 세 부분으로 구성됩니다:

* **프롬프트**: 현실적인 사용자 메시지 — 누군가가 실제로 입력할 법한 것.
* **예상 출력**: 성공이 어떤 모습인지에 대한 사람이 읽을 수 있는 설명.
* **입력 파일** (선택사항): 스킬이 작업하는 데 필요한 파일.

스킬 디렉토리 내 `evals/evals.json`에 테스트 케이스를 저장하세요:

```json evals/evals.json theme={null}
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "data/sales_2025.csv에 월별 판매 데이터 CSV가 있어. 매출 기준 상위 3개월을 찾아서 막대 차트를 만들어줄 수 있어?",
      "expected_output": "레이블된 축과 값이 있는 매출 기준 상위 3개월을 보여주는 막대 차트 이미지.",
      "files": ["evals/files/sales_2025.csv"]
    },
    {
      "id": 2,
      "prompt": "다운로드 폴더에 customers.csv가 있는데, 일부 행에 이메일이 없어 — 정리해주고 몇 개가 없었는지 알려줄 수 있어?",
      "expected_output": "누락된 이메일이 처리된 정리된 CSV와 몇 개가 누락되었는지 개수.",
      "files": ["evals/files/customers.csv"]
    }
  ]
}
```

**좋은 테스트 프롬프트 작성 팁:**

* **2-3개의 테스트 케이스로 시작하세요.** 첫 번째 결과를 보기 전에 과도하게 투자하지 마세요. 나중에 세트를 확장할 수 있습니다.
* **프롬프트를 다양하게 변형하세요.** 다른 표현, 세부 수준, 격식을 사용하세요. 일부 프롬프트는 일상적이어야 하고 ("이 csv 좀 정리해줄 수 있어"), 다른 것들은 정확해야 합니다 ("data/input.csv에 있는 CSV를 파싱하고, B열이 null인 행을 제거하고, 결과를 data/output.csv에 저장해줘").
* **엣지 케이스를 커버하세요.** 경계 조건을 테스트하는 프롬프트를 최소 하나 포함하세요 — 잘못된 형식의 입력, 비정상적인 요청, 또는 스킬의 지시가 모호할 수 있는 경우.
* **현실적인 맥락을 사용하세요.** 실제 사용자는 파일 경로, 열 이름, 개인적 맥락을 언급합니다. "이 데이터 처리해줘"와 같은 프롬프트는 유용한 것을 테스트하기에 너무 모호합니다.

아직 구체적인 합격/불합격 검사를 정의하는 것에 대해 걱정하지 마세요 — 프롬프트와 예상 출력만 있으면 됩니다. 첫 번째 실행이 생성하는 것을 본 후에 상세 검사(어서션이라고 함)를 추가할 것입니다.

## 평가 실행하기

핵심 패턴은 각 테스트 케이스를 두 번 실행하는 것입니다: 한 번은 **스킬과 함께**, 한 번은 **스킬 없이** (또는 이전 버전으로). 이것은 비교할 기준선을 제공합니다.

### 워크스페이스 구조

스킬 디렉토리 옆 워크스페이스 디렉토리에 평가 결과를 구성하세요. 전체 평가 루프를 한 번 통과할 때마다 자체 `iteration-N/` 디렉토리를 가집니다. 그 안에서 각 테스트 케이스는 `with_skill/`과 `without_skill/` 하위 디렉토리가 있는 평가 디렉토리를 가집니다:

```
csv-analyzer/
├── SKILL.md
└── evals/
    └── evals.json
csv-analyzer-workspace/
└── iteration-1/
    ├── eval-top-months-chart/
    │   ├── with_skill/
    │   │   ├── outputs/       # 실행으로 생성된 파일
    │   │   ├── timing.json    # 토큰 및 소요 시간
    │   │   └── grading.json   # 어서션 결과
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    ├── eval-clean-missing-emails/
    │   ├── with_skill/
    │   │   ├── outputs/
    │   │   ├── timing.json
    │   │   └── grading.json
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    └── benchmark.json         # 집계 통계
```

직접 작성하는 주요 파일은 `evals/evals.json`입니다. 다른 JSON 파일들 (`grading.json`, `timing.json`, `benchmark.json`)은 평가 과정 중에 — 에이전트, 스크립트, 또는 사용자에 의해 — 생성됩니다.

### 실행 생성하기

각 평가 실행은 깨끗한 컨텍스트로 시작해야 합니다 — 이전 실행이나 스킬 개발 과정의 잔여 상태가 없어야 합니다. 이것은 에이전트가 `SKILL.md`가 지시하는 것만 따르도록 보장합니다. 서브에이전트를 지원하는 환경(예: Claude Code)에서는 이 격리가 자연스럽게 이루어집니다: 각 자식 태스크는 새로 시작합니다. 서브에이전트 없이는 각 실행에 별도의 세션을 사용하세요.

각 실행에 대해 제공하세요:

* 스킬 경로 (기준선의 경우 스킬 없음)
* 테스트 프롬프트
* 입력 파일
* 출력 디렉토리

다음은 단일 with-skill 실행을 위해 에이전트에게 줄 지시 예시입니다:

```
이 태스크를 실행하세요:
- 스킬 경로: /path/to/csv-analyzer
- 태스크: data/sales_2025.csv에 월별 판매 데이터 CSV가 있어.
  매출 기준 상위 3개월을 찾아서 막대 차트를 만들어줄 수 있어?
- 입력 파일: evals/files/sales_2025.csv
- 출력 저장 위치: csv-analyzer-workspace/iteration-1/eval-top-months-chart/with_skill/outputs/
```

기준선의 경우, 스킬 경로 없이 동일한 프롬프트를 사용하고 `without_skill/outputs/`에 저장하세요.

기존 스킬을 개선할 때는 이전 버전을 기준선으로 사용하세요. 편집 전에 스냅샷을 찍고 (`cp -r <skill-path> <workspace>/skill-snapshot/`), 기준선 실행이 스냅샷을 가리키도록 하고, `without_skill/outputs/` 대신 `old_skill/outputs/`에 저장하세요.

### 타이밍 데이터 캡처하기

타이밍 데이터를 통해 스킬이 기준선 대비 얼마나 많은 시간과 토큰을 소비하는지 비교할 수 있습니다 — 출력 품질을 크게 개선하지만 토큰 사용량을 세 배로 늘리는 스킬은 더 나으면서 더 저렴한 스킬과는 다른 트레이드오프입니다. 각 실행이 완료되면 토큰 수와 소요 시간을 기록하세요:

```json timing.json theme={null}
{
  "total_tokens": 84852,
  "duration_ms": 23332
}
```

<Tip>
  Claude Code에서 서브에이전트 태스크가 완료되면, [태스크 완료 알림](https://platform.claude.com/docs/en/agent-sdk/typescript#sdk-task-notification-message)에 `total_tokens`와 `duration_ms`가 포함됩니다. 이 값들은 다른 곳에 저장되지 않으므로 즉시 저장하세요.
</Tip>

## 어서션 작성하기

어서션은 출력이 포함해야 하거나 달성해야 하는 것에 대해 검증 가능한 진술입니다. 첫 번째 라운드의 출력을 본 후에 추가하세요 — 스킬이 실행되기 전까지는 "좋은" 것이 어떤 모습인지 모르는 경우가 많습니다.

좋은 어서션:

* `"출력 파일이 유효한 JSON입니다"` — 프로그래밍 방식으로 검증 가능.
* `"막대 차트에 레이블된 축이 있습니다"` — 구체적이고 관찰 가능.
* `"보고서에 최소 3개의 권장사항이 포함됩니다"` — 셀 수 있음.

약한 어서션:

* `"출력이 좋습니다"` — 채점하기에 너무 모호함.
* `"출력에 정확히 'Total Revenue: $X'라는 문구가 사용됩니다"` — 너무 취약함; 다른 표현으로 올바른 출력이 실패함.

모든 것에 어서션이 필요하지는 않습니다. 일부 품질 — 작성 스타일, 시각적 디자인, 출력이 "맞는 느낌"인지 여부 — 은 합격/불합격 검사로 분해하기 어렵습니다. 이것들은 [인간 검토](#인간과-함께-결과-검토하기) 중에 더 잘 잡힙니다. 어서션은 객관적으로 확인할 수 있는 것들을 위해 남겨두세요.

`evals/evals.json`의 각 테스트 케이스에 어서션을 추가하세요:

```json evals/evals.json highlight={9-14} theme={null}
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "data/sales_2025.csv에 월별 판매 데이터 CSV가 있어. 매출 기준 상위 3개월을 찾아서 막대 차트를 만들어줄 수 있어?",
      "expected_output": "레이블된 축과 값이 있는 매출 기준 상위 3개월을 보여주는 막대 차트 이미지.",
      "files": ["evals/files/sales_2025.csv"],
      "assertions": [
        "출력에 막대 차트 이미지 파일이 포함됩니다",
        "차트가 정확히 3개월을 보여줍니다",
        "두 축 모두 레이블이 있습니다",
        "차트 제목이나 캡션에 매출이 언급됩니다"
      ]
    }
  ]
}
```

## 출력 채점하기

채점은 실제 출력에 대해 각 어서션을 평가하고 구체적인 증거와 함께 **PASS** 또는 **FAIL**을 기록하는 것을 의미합니다. 증거는 단순히 의견을 진술하는 것이 아니라 출력을 인용하거나 참조해야 합니다.

가장 간단한 접근법은 출력과 어서션을 LLM에 제공하고 각각을 평가하도록 요청하는 것입니다. 코드로 확인할 수 있는 어서션(유효한 JSON, 올바른 행 수, 예상 차원의 파일 존재)의 경우 검증 스크립트를 사용하세요 — 스크립트는 기계적인 검사에서 LLM 판단보다 더 신뢰할 수 있고 반복 간에 재사용 가능합니다.

```json grading.json theme={null}
{
  "assertion_results": [
    {
      "text": "출력에 막대 차트 이미지 파일이 포함됩니다",
      "passed": true,
      "evidence": "출력 디렉토리에서 chart.png (45KB)를 찾았습니다"
    },
    {
      "text": "차트가 정확히 3개월을 보여줍니다",
      "passed": true,
      "evidence": "차트에 3월, 7월, 11월의 막대가 표시됩니다"
    },
    {
      "text": "두 축 모두 레이블이 있습니다",
      "passed": false,
      "evidence": "Y축에는 'Revenue ($)' 레이블이 있지만 X축에는 레이블이 없습니다"
    },
    {
      "text": "차트 제목이나 캡션에 매출이 언급됩니다",
      "passed": true,
      "evidence": "차트 제목이 'Top 3 Months by Revenue'입니다"
    }
  ],
  "summary": {
    "passed": 3,
    "failed": 1,
    "total": 4,
    "pass_rate": 0.75
  }
}
```

### 채점 원칙

* **PASS에는 구체적인 증거를 요구하세요.** 의심스러울 때는 이익을 주지 마세요. 어서션이 "요약을 포함한다"고 하고 출력에 모호한 한 문장의 "요약"이라는 섹션이 있다면, 그것은 FAIL입니다 — 레이블은 있지만 내용이 없습니다.
* **결과만이 아닌 어서션 자체를 검토하세요.** 채점하는 동안, 어서션이 너무 쉬운지 (스킬 품질에 관계없이 항상 통과), 너무 어려운지 (출력이 좋더라도 항상 실패), 또는 검증 불가능한지 (출력만으로는 확인할 수 없음) 주목하세요. 다음 반복을 위해 이것들을 수정하세요.

<Tip>
  두 가지 스킬 버전을 비교할 때 **맹목 비교**를 시도해 보세요: 어떤 버전에서 왔는지 공개하지 않고 두 출력을 LLM 판사에게 제시합니다. 판사는 어떤 버전이 "더 나아야 한다"는 편견 없이 자체 기준으로 구성, 형식, 유용성, 완성도와 같은 전체적 품질을 채점합니다. 이것은 어서션 채점을 보완합니다: 두 출력이 모든 어서션을 통과하더라도 전체적인 품질에서 크게 다를 수 있습니다.
</Tip>

## 결과 집계하기

반복의 모든 실행이 채점되면, 구성별 요약 통계를 계산하고 평가 디렉토리 옆 `benchmark.json`에 저장하세요 (예: `csv-analyzer-workspace/iteration-1/benchmark.json`):

```json benchmark.json theme={null}
{
  "run_summary": {
    "with_skill": {
      "pass_rate": { "mean": 0.83, "stddev": 0.06 },
      "time_seconds": { "mean": 45.0, "stddev": 12.0 },
      "tokens": { "mean": 3800, "stddev": 400 }
    },
    "without_skill": {
      "pass_rate": { "mean": 0.33, "stddev": 0.10 },
      "time_seconds": { "mean": 32.0, "stddev": 8.0 },
      "tokens": { "mean": 2100, "stddev": 300 }
    },
    "delta": {
      "pass_rate": 0.50,
      "time_seconds": 13.0,
      "tokens": 1700
    }
  }
}
```

`delta`는 스킬이 무엇을 비용으로 치르는지 (더 많은 시간, 더 많은 토큰)와 무엇을 얻는지 (더 높은 통과율)를 알려줍니다. 13초를 더 사용하지만 통과율을 50퍼센트 포인트 향상시키는 스킬은 아마 그만한 가치가 있습니다. 토큰 사용량을 두 배로 늘리고 2포인트 개선하는 스킬은 그렇지 않을 수 있습니다.

<Note>
  표준 편차(`stddev`)는 평가당 여러 번 실행할 때만 의미가 있습니다. 2-3개의 테스트 케이스와 단일 실행으로 초기 반복에서는 원시 통과 수와 델타에 집중하세요 — 테스트 세트를 확장하고 각 평가를 여러 번 실행할 때 통계 수치가 유용해집니다.
</Note>

## 패턴 분석하기

집계 통계는 중요한 패턴을 숨길 수 있습니다. 벤치마크를 계산한 후:

* **두 구성 모두에서 항상 통과하는 어서션을 제거하거나 교체하세요.** 이것들은 아무것도 유용하게 알려주지 않습니다 — 모델이 스킬 없이도 잘 처리합니다. 실제 스킬 가치를 반영하지 않고 with-skill 통과율을 부풀립니다.
* **두 구성 모두에서 항상 실패하는 어서션을 조사하세요.** 어서션이 잘못되었거나 (모델이 할 수 없는 것을 요구), 테스트 케이스가 너무 어렵거나, 어서션이 잘못된 것을 확인하고 있습니다. 다음 반복 전에 이것들을 수정하세요.
* **스킬과 함께는 통과하지만 없이는 실패하는 어서션을 연구하세요.** 여기서 스킬이 명확히 가치를 추가하고 있습니다. *왜*를 이해하세요 — 어떤 지시나 스크립트가 차이를 만들었나요?
* **실행 간에 결과가 일관되지 않을 때 지시를 강화하세요.** 동일한 평가가 때로는 통과하고 때로는 실패한다면 (벤치마크에서 높은 `stddev`로 반영됨), 평가가 불안정하거나 (모델 무작위성에 민감), 스킬의 지시가 모델이 매번 다르게 해석할 만큼 모호할 수 있습니다. 모호성을 줄이기 위해 예시나 더 구체적인 지침을 추가하세요.
* **시간 및 토큰 이상값을 확인하세요.** 하나의 평가가 다른 것들보다 3배 더 오래 걸린다면, 실행 기록 (실행 중 모델이 한 작업의 전체 로그)을 읽어 병목을 찾으세요.

## 인간과 함께 결과 검토하기

어서션 채점과 패턴 분석은 많은 것을 잡아내지만, 어서션을 작성하면서 생각했던 것만 확인합니다. 인간 검토자는 새로운 시각을 가져옵니다 — 예상하지 못했던 문제를 잡아내고, 출력이 기술적으로 올바르지만 핵심을 놓칠 때 알아차리고, 합격/불합격 검사로 표현하기 어려운 문제를 발견합니다. 각 테스트 케이스에 대해 채점과 함께 실제 출력을 검토하세요.

각 테스트 케이스에 대한 구체적인 피드백을 기록하고 워크스페이스에 저장하세요 (예: 평가 디렉토리 옆 `feedback.json`으로):

```json feedback.json theme={null}
{
  "eval-top-months-chart": "차트에 축 레이블이 없고 월이 날짜순 대신 알파벳순으로 정렬되어 있습니다.",
  "eval-clean-missing-emails": ""
}
```

"차트에 축 레이블이 없습니다"는 실행 가능하고, "나빠 보인다"는 그렇지 않습니다. 빈 피드백은 출력이 괜찮아 보였다는 의미입니다 — 해당 테스트 케이스가 검토를 통과했습니다. [반복 단계](#스킬-반복하기)에서 구체적인 불만이 있었던 테스트 케이스에 개선에 집중하세요.

## 스킬 반복하기

채점과 검토 후, 세 가지 신호 소스가 있습니다:

* **실패한 어서션**은 구체적인 격차를 가리킵니다 — 누락된 단계, 불명확한 지시, 또는 스킬이 처리하지 못하는 경우.
* **인간 피드백**은 더 광범위한 품질 문제를 가리킵니다 — 접근법이 잘못되었거나, 출력이 구조가 나쁘거나, 스킬이 기술적으로 올바르지만 도움이 되지 않는 결과를 생성했습니다.
* **실행 기록**은 *왜* 일이 잘못되었는지 드러냅니다. 에이전트가 지시를 무시했다면, 지시가 모호할 수 있습니다. 에이전트가 비생산적인 단계에 시간을 낭비했다면, 그 지시들은 단순화하거나 제거해야 할 수 있습니다.

이러한 신호를 스킬 개선으로 전환하는 가장 효과적인 방법은 세 가지 모두 — 현재 `SKILL.md`와 함께 — LLM에 제공하고 변경 사항을 제안하도록 요청하는 것입니다. LLM은 수동으로 연결하기 번거로운 실패한 어서션, 검토자 불만, 기록 동작 전반에 걸친 패턴을 합성할 수 있습니다. LLM에 프롬프트할 때 다음 지침을 포함하세요:

* **피드백에서 일반화하세요.** 스킬은 테스트 케이스만이 아닌 다양한 프롬프트에 걸쳐 사용될 것입니다. 수정은 특정 예시에 대한 좁은 패치를 추가하는 것보다 근본적인 문제를 광범위하게 다루어야 합니다.
* **스킬을 간결하게 유지하세요.** 적지만 더 나은 지시가 포괄적인 규칙보다 성능이 더 좋은 경우가 많습니다. 기록에 낭비된 작업이 보이면 (불필요한 검증, 필요 없는 중간 출력), 해당 지시를 제거하세요. 더 많은 규칙을 추가해도 통과율이 정체된다면, 스킬이 과도하게 제약될 수 있습니다 — 지시를 제거하고 결과가 유지되거나 개선되는지 확인하세요.
* **이유를 설명하세요.** 추론 기반 지시 ("Y가 Z를 유발하는 경향이 있기 때문에 X를 수행하라")가 엄격한 지시 ("항상 X를 수행하라, 절대 Y를 하지 말라")보다 잘 작동합니다. 모델은 목적을 이해할 때 지시를 더 안정적으로 따릅니다.
* **반복적인 작업을 번들링하세요.** 모든 테스트 실행에서 독립적으로 유사한 헬퍼 스크립트 (차트 빌더, 데이터 파서)를 작성했다면, 그것은 스킬의 `scripts/` 디렉토리에 한 번 번들링할 신호입니다. 이것을 수행하는 방법은 [스크립트 사용하기](/skill-creation/using-scripts)를 참조하세요.

### 루프

1. 평가 신호와 현재 `SKILL.md`를 LLM에 제공하고 개선 사항을 제안하도록 요청합니다.
2. 변경 사항을 검토하고 적용합니다.
3. 새 `iteration-<N+1>/` 디렉토리에서 모든 테스트 케이스를 다시 실행합니다.
4. 새 결과를 채점하고 집계합니다.
5. 인간과 함께 검토합니다. 반복합니다.

결과에 만족하거나, 피드백이 일관되게 비어있거나, 반복 간에 더 이상 의미 있는 개선이 보이지 않을 때 중단하세요.
