# 스킬 설명 최적화하기

> 관련 프롬프트에서 스킬이 안정적으로 트리거되도록 설명을 개선하는 방법.

스킬은 활성화될 때만 도움이 됩니다. `SKILL.md` 프론트매터의 `description` 필드는 에이전트가 주어진 작업에 스킬을 로드할지 결정할 때 사용하는 주요 메커니즘입니다. 설명이 부족하면 스킬이 필요할 때 트리거되지 않고, 너무 광범위한 설명은 불필요할 때 트리거됩니다.

이 가이드는 트리거 정확도를 위해 스킬의 설명을 체계적으로 테스트하고 개선하는 방법을 다룹니다.

## 스킬 트리거 작동 방식

에이전트는 [점진적 공개](/specification#progressive-disclosure)를 사용하여 컨텍스트를 관리합니다. 시작 시 각 사용 가능한 스킬의 `name`과 `description`만 로드합니다 — 스킬이 언제 관련될 수 있는지 결정하기에 충분한 정보만 가져옵니다. 사용자의 작업이 설명과 일치하면, 에이전트는 전체 `SKILL.md`를 컨텍스트에 읽어들이고 지시를 따릅니다.

이는 설명이 트리거의 전체 부담을 지고 있음을 의미합니다. 설명이 스킬이 언제 유용한지 전달하지 못하면, 에이전트는 스킬을 사용해야 할 때를 알지 못합니다.

한 가지 중요한 뉘앙스: 에이전트는 일반적으로 스스로 처리할 수 없는 지식이나 기능이 필요한 작업에만 스킬을 참조합니다. "이 PDF 읽어줘"처럼 간단한 한 단계 요청은 설명이 완벽하게 일치하더라도 PDF 스킬을 트리거하지 않을 수 있습니다. 에이전트가 기본 도구로 처리할 수 있기 때문입니다. 잘 작성된 설명이 차이를 만드는 것은 특화된 지식이 필요한 작업 — 낯선 API, 도메인 특화 워크플로우, 또는 흔치 않은 형식 — 에서입니다.

## 효과적인 설명 작성하기

테스트 전에, 좋은 설명이 어떤 모습인지 아는 것이 도움이 됩니다. 몇 가지 원칙:

* **명령형 표현 사용.** 설명을 에이전트에게 보내는 지시로 구성하세요: "이 스킬을 사용할 때는..." 에이전트는 행동할지 결정하는 중이므로, 언제 행동해야 하는지 알려주세요.
* **구현이 아닌 사용자 의도에 집중.** 스킬의 내부 메커니즘이 아닌 사용자가 달성하려는 것을 설명하세요. 에이전트는 사용자가 요청한 내용에 대해 매칭합니다.
* **적극적인 표현 쪽으로 치우치세요.** 사용자가 도메인을 직접 언급하지 않는 경우를 포함하여 스킬이 적용되는 맥락을 명시적으로 나열하세요: "사용자가 'CSV'나 '분석'을 명시적으로 언급하지 않더라도."
* **간결하게 유지하세요.** 몇 문장에서 짧은 단락이 보통 적당합니다 — 스킬의 범위를 커버하기에 충분히 길고, 여러 스킬에 걸쳐 에이전트의 컨텍스트를 부풀리지 않을 만큼 충분히 짧습니다. [명세서](/specification#description-field)는 1024자의 상한선을 강제합니다.

## 트리거 평가 쿼리 설계하기

트리거를 테스트하려면 평가 쿼리 세트가 필요합니다 — 스킬을 트리거해야 하는지 여부가 레이블된 현실적인 사용자 프롬프트입니다.

```json eval_queries.json theme={null}
[
  { "query": "~/data/q4_results.xlsx에 있는 스프레드시트에서 C열이 매출이고 D열이 지출인데 — 이익률 열을 추가하고 10% 미만인 것을 강조 표시해줄 수 있어?", "should_trigger": true },
  { "query": "이 json 파일을 yaml로 변환하는 가장 빠른 방법이 뭐야", "should_trigger": false }
]
```

약 20개의 쿼리를 목표로 하세요: 트리거해야 하는 것 8-10개, 트리거하지 않아야 하는 것 8-10개.

### 트리거해야 하는 쿼리

이것들은 설명이 스킬의 범위를 파악하는지 테스트합니다. 여러 축을 따라 다양하게 변형하세요:

* **표현 방식**: 공식적인 것, 일상적인 것, 오타나 약어가 있는 것.
* **명시성**: 스킬의 도메인을 직접 명명하는 것("이 CSV 분석해줘"), 직접 명명하지 않고 필요를 설명하는 것("상사가 이 데이터 파일로 차트를 원해").
* **세부 수준**: 간결한 프롬프트와 맥락이 풍부한 것을 혼합 — 짧은 "내 판매 CSV 분석하고 차트 만들어줘"와 파일 경로, 열 이름, 배경 이야기가 있는 더 긴 메시지.
* **복잡성**: 단계 수와 결정 지점을 다양하게. 에이전트가 작업이 더 큰 체인에 묻혀 있을 때 스킬이 관련된다고 파악할 수 있는지 테스트하기 위해 단일 단계 작업과 다단계 워크플로우를 포함하세요.

가장 유용한 트리거 쿼리는 스킬이 도움이 되지만 쿼리만으로는 연결이 명확하지 않은 것들입니다. 이런 경우가 설명 표현이 차이를 만드는 상황입니다 — 쿼리가 이미 스킬이 하는 것을 정확히 요청하면, 어떤 합리적인 설명도 트리거됩니다.

### 트리거하지 않아야 하는 쿼리

가장 가치 있는 부정 테스트 케이스는 **근접 실패** — 스킬과 키워드나 개념을 공유하지만 실제로는 다른 것이 필요한 쿼리입니다. 이것들은 설명이 단순히 광범위한 것이 아닌 정확한지 테스트합니다.

CSV 분석 스킬의 경우, 약한 부정 예시:

* `"피보나치 함수 작성해줘"` — 명백히 관련 없음, 아무것도 테스트하지 않음.
* `"오늘 날씨가 어때?"` — 키워드 겹침 없음, 너무 쉬움.

강한 부정 예시:

* `"내 Excel 예산 스프레드시트의 수식을 업데이트해야 해"` — "스프레드시트"와 "데이터" 개념을 공유하지만, CSV 분석이 아닌 Excel 편집이 필요함.
* `"csv를 읽고 각 행을 postgres 데이터베이스에 업로드하는 python 스크립트를 작성해줄 수 있어"` — CSV를 포함하지만, 작업은 분석이 아닌 데이터베이스 ETL임.

### 현실감을 위한 팁

실제 사용자 프롬프트에는 일반적인 테스트 쿼리에 없는 맥락이 포함됩니다. 다음을 포함하세요:

* 파일 경로 (`~/Downloads/report_final_v2.xlsx`)
* 개인적 맥락 (`"매니저가 나에게..."`)
* 구체적인 세부 사항 (열 이름, 회사 이름, 데이터 값)
* 일상 언어, 약어, 간헐적인 오타

## 설명이 트리거되는지 테스트하기

기본 접근법: 스킬이 설치된 상태에서 에이전트를 통해 각 쿼리를 실행하고 에이전트가 스킬을 호출하는지 관찰합니다. 스킬이 에이전트에 등록되어 발견 가능한지 확인하세요 — 이것이 어떻게 작동하는지는 클라이언트마다 다릅니다 (예: 스킬 디렉토리, 설정 파일, 또는 CLI 플래그).

대부분의 에이전트 클라이언트는 실행 로그, 도구 호출 이력, 또는 상세 출력과 같은 일종의 관측 가능성을 제공합니다 — 실행 중 어떤 스킬이 참조되었는지 볼 수 있게 해줍니다. 자세한 내용은 클라이언트 문서를 확인하세요. 에이전트가 스킬의 `SKILL.md`를 로드했다면 스킬이 트리거된 것이고, 스킬을 참조하지 않고 진행했다면 트리거되지 않은 것입니다.

쿼리가 "통과"하는 경우:

* `should_trigger`가 `true`이고 스킬이 호출되었거나,
* `should_trigger`가 `false`이고 스킬이 호출되지 않은 경우.

### 여러 번 실행하기

모델 동작은 비결정론적입니다 — 동일한 쿼리가 한 번은 스킬을 트리거하고 다음번에는 그렇지 않을 수 있습니다. 각 쿼리를 여러 번 실행하고 (3번이 합리적인 시작점) **트리거율**을 계산하세요: 스킬이 호출된 실행의 비율.

트리거해야 하는 쿼리는 트리거율이 임계값 이상이면 통과합니다 (0.5가 합리적인 기본값). 트리거하지 않아야 하는 쿼리는 트리거율이 해당 임계값 미만이면 통과합니다.

3번 실행에 20개 쿼리면 60번의 호출이 됩니다. 스크립트를 작성하는 것이 좋습니다. 다음은 일반적인 구조입니다 — `claude` 호출과 `check_triggered`의 감지 로직을 에이전트 클라이언트가 제공하는 것으로 교체하세요:

```bash theme={null}
#!/bin/bash
QUERIES_FILE="${1:?Usage: $0 <queries.json>}"
SKILL_NAME="my-skill"
RUNS=3

# 이 예시는 Claude Code의 JSON 출력을 사용하여 Skill 도구 호출을 확인합니다.
# 이 함수를 에이전트 클라이언트의 감지 로직으로 교체하세요.
# 스킬이 호출된 경우 0(성공)을 반환하고, 그렇지 않으면 1을 반환해야 합니다.
check_triggered() {
  local query="$1"
  claude -p "$query" --output-format json 2>/dev/null \
    | jq -e --arg skill "$SKILL_NAME" \
      'any(.messages[].content[]; .type == "tool_use" and .name == "Skill" and .input.skill == $skill)' \
      > /dev/null 2>&1
}

count=$(jq length "$QUERIES_FILE")
for i in $(seq 0 $((count - 1))); do
  query=$(jq -r ".[$i].query" "$QUERIES_FILE")
  should_trigger=$(jq -r ".[$i].should_trigger" "$QUERIES_FILE")
  triggers=0

  for run in $(seq 1 $RUNS); do
    check_triggered "$query" && triggers=$((triggers + 1))
  done

  jq -n \
    --arg query "$query" \
    --argjson should_trigger "$should_trigger" \
    --argjson triggers "$triggers" \
    --argjson runs "$RUNS" \
    '{query: $query, should_trigger: $should_trigger, triggers: $triggers, runs: $runs, trigger_rate: ($triggers / $runs)}'
done | jq -s '.'
```

<Tip>
  에이전트 클라이언트가 지원한다면, 결과가 명확해지면 실행을 일찍 중단할 수 있습니다 — 에이전트가 스킬을 참조했거나 참조하지 않고 작업을 시작했거나. 이것은 전체 평가 세트를 실행하는 시간과 비용을 크게 줄일 수 있습니다.
</Tip>

## 훈련/검증 분할로 과적합 방지하기

모든 쿼리에 대해 설명을 최적화하면 과적합 위험이 있습니다 — 이 특정 표현에는 작동하지만 새로운 표현에서는 실패하는 설명을 만들게 됩니다.

해결책은 쿼리 세트를 분할하는 것입니다:

* **훈련 세트 (~60%)**: 실패를 식별하고 개선을 안내하는 데 사용하는 쿼리.
* **검증 세트 (~40%)**: 따로 보관해두고 개선 사항이 일반화되는지 확인할 때만 사용하는 쿼리.

두 세트 모두 트리거해야 하는 쿼리와 그렇지 않아야 하는 쿼리의 비율적 혼합을 포함하도록 하세요 — 모든 긍정 케이스가 한 세트에 들어가지 않도록 주의하세요. 무작위로 섞고 반복 간 비교가 사과 대 사과가 되도록 분할을 고정하세요.

[위](#여러-번-실행하기)와 같은 스크립트를 사용하는 경우, 쿼리를 두 파일 — `train_queries.json`과 `validation_queries.json` — 로 분할하고 각각에 대해 스크립트를 별도로 실행할 수 있습니다.

## 최적화 루프

1. **평가** — *훈련 세트와 검증 세트* 모두에서 현재 설명을 평가합니다. 훈련 결과는 변경 사항을 안내하고, 검증 결과는 그 변경 사항이 일반화되는지 알려줍니다.
2. **실패 식별** — *훈련 세트*에서: 어떤 트리거해야 하는 쿼리가 트리거되지 않았나? 어떤 트리거하지 않아야 하는 쿼리가 트리거되었나?
   * 설명을 직접 수정하든 LLM에게 프롬프트하든, 변경 사항을 안내하는 데는 훈련 세트 실패만 사용하고 검증 세트 결과는 프로세스에서 배제하세요.
3. **설명 수정.** 일반화에 집중하세요:
   * 트리거해야 하는 쿼리가 실패하는 경우, 설명이 너무 좁을 수 있습니다. 범위를 넓히거나 스킬이 언제 유용한지에 대한 맥락을 추가하세요.
   * 트리거하지 않아야 하는 쿼리가 거짓 트리거되는 경우, 설명이 너무 광범위할 수 있습니다. 스킬이 *하지 않는* 것에 대한 구체성을 추가하거나 이 스킬과 인접 기능 간의 경계를 명확히 하세요.
   * 실패한 쿼리에서 특정 키워드를 추가하는 것은 피하세요 — 과적합입니다. 대신 해당 쿼리들이 나타내는 일반적인 범주나 개념을 찾아 그것을 다루세요.
   * 여러 번 반복 후에도 막힌다면, 점진적 조정보다는 설명에 구조적으로 다른 접근법을 시도해 보세요. 다른 프레이밍이나 문장 구조가 조정으로는 돌파할 수 없는 문제를 해결할 수 있습니다.
   * 설명이 1024자 제한 아래에 있는지 확인하세요 — 설명은 최적화 중에 길어지는 경향이 있습니다.
4. **반복** — 모든 *훈련 세트* 쿼리가 통과하거나 의미 있는 개선이 보이지 않을 때까지 1-3단계를 반복합니다.
5. **최상의 반복 선택** — *검증 세트*의 통과율로 선택합니다 — 검증 세트에서 통과한 쿼리의 비율. 최상의 설명이 마지막에 생성한 것이 아닐 수 있음을 유의하세요. 이전 반복이 훈련 세트에 과적합된 이후 반복보다 더 높은 검증 통과율을 가질 수 있습니다.

5번의 반복이 보통 충분합니다. 성능이 개선되지 않는다면, 문제가 설명보다는 쿼리 (너무 쉬움, 너무 어려움, 또는 잘못 레이블됨)에 있을 수 있습니다.

<Tip>
  [`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator) 스킬은 이 루프를 종단간 자동화합니다: 평가 세트를 분할하고, 병렬로 트리거율을 평가하고, Claude를 사용하여 설명 개선을 제안하고, 실행 중에 볼 수 있는 실시간 HTML 보고서를 생성합니다.
</Tip>

## 결과 적용하기

최상의 설명을 선택했다면:

1. `SKILL.md` 프론트매터의 `description` 필드를 업데이트합니다.
2. 설명이 [1024자 제한](/specification#description-field) 아래에 있는지 확인합니다.
3. 설명이 예상대로 트리거되는지 확인합니다. 빠른 정상 확인을 위해 몇 가지 프롬프트를 수동으로 시도해 보세요. 더 엄격한 테스트를 위해 5-10개의 새로운 쿼리 (트리거해야 하는 것과 그렇지 않아야 하는 것의 혼합)를 작성하고 평가 스크립트로 실행하세요 — 이 쿼리들은 최적화 과정의 일부가 아니었으므로, 설명이 일반화되는지에 대한 솔직한 확인을 제공합니다.

개선 전후 비교:

```yaml theme={null}
# 개선 전
description: CSV 파일을 처리합니다.

# 개선 후
description: >
  CSV 및 표 형식 데이터 파일 분석 — 요약 통계 계산,
  파생 열 추가, 차트 생성, 지저분한 데이터 정리. 사용자가
  CSV, TSV, 또는 Excel 파일을 가지고 데이터를 탐색, 변환,
  또는 시각화하려 할 때 이 스킬을 사용하세요. 사용자가
  "CSV"나 "분석"을 명시적으로 언급하지 않더라도.
```

개선된 설명은 스킬이 하는 것에 대해 더 구체적이고 (요약 통계, 파생 열, 차트, 정리), 적용되는 시점에 대해 더 광범위합니다 (CSV, TSV, Excel; 명시적 키워드 없이도).
