정규화를 마친 데이터베이스가 실제 시스템에서 어떻게 활용되는지 이해하려면, 서비스의 목적에 따른 두 가지 거대한 흐름인 **OLTP**와 **OLAP**를 반드시 알아야 합니다.

데이터 모델링이 '설계도'라면, OLTP와 OLAP는 그 설계도가 실제로 '어떤 용도의 건물'에 쓰이는지를 결정합니다.

---

## 1. OLTP (Online Transaction Processing)

**"현재 발생하고 있는 거래를 빠르고 정확하게 처리하는 것"**

OLTP는 우리가 일상적으로 사용하는 대부분의 앱이나 웹 서비스의 뒷단에서 작동합니다. 은행 송금, 상품 주문, 수강 신청 등이 대표적입니다.

* **핵심 목표:** 무결성 보장과 빠른 응답 속도 (Read/Write의 균형).
* **모델링 특징:** 고도의 정규화(Normalization)를 지향합니다.
* 중복 데이터를 최소화하여 **데이터 무결성**을 지키고, 삽입/수정/삭제 시 발생하는 이상 현상을 방지합니다.
* 테이블을 잘게 쪼개어 트랜잭션의 단위를 작게 유지합니다.


* **비유:** 편의점의 계산대. 물건이 팔릴 때마다 재고를 줄이고 결제 내역을 실시간으로 정확하게 기록해야 합니다.

---

## 2. OLAP (Online Analytical Processing)

**"누적된 대량의 데이터를 분석하여 의사결정에 활용하는 것"**

OLAP는 경영진의 리포트, 마케팅 타겟 추출, 연간 매출 트렌드 분석 등에 사용됩니다. 주로 데이터 웨어하우스(DW) 환경에서 운영됩니다.

* **핵심 목표:** 복잡한 질의(Query)에 대한 빠른 조회 속도 (Read 위주).
* **모델링 특징:** 반정규화(De-normalization)를 적극적으로 활용합니다.
* 정규화된 테이블을 다시 합쳐서 **조인(Join) 횟수를 줄입니다.** 수억 건의 데이터를 분석할 때 조인이 많으면 성능이 급격히 떨어지기 때문입니다.
* 주로 **스타 스키마(Star Schema)** 형태를 띱니다.


* **비유:** 편의점 본사의 연간 매출 분석 보고서. 개별 결제 건보다는 "지난 1년간 전국에서 가장 많이 팔린 물품 10개" 같은 통계값이 중요합니다.

---

## 3. 정규화와 OLTP/OLAP의 관계 (비교 테이블)

| 구분 | OLTP (트랜잭션 중심) | OLAP (분석 중심) |
| --- | --- | --- |
| **주요 작업** | Insert, Update, Delete (쓰기 위주) | Select (읽기 위주) |
| **데이터 모델** | **정규화된 모델 (3NF, BCNF)** | **반정규화된 모델 (Star Schema)** |
| **데이터 단위** | 개별 트랜잭션 (Row 단위) | 대량의 이력 데이터 (Column/Batch) |
| **정규화 이유** | 데이터 중복 제거 및 무결성 유지 | (정규화보다 성능 우선) 조인 최소화 |
| **저장 장소** | 운영 데이터베이스 (Operational DB) | 데이터 웨어하우스 (DW) |

---

## 4. 실제 시스템에서의 적용 흐름

실제 기업용 시스템은 이 두 가지를 분리하여 운영하는 것이 일반적입니다.

1. **운영 단계 (OLTP):** 정규화가 잘 된 DB에서 실시간으로 주문을 받고 회원 정보를 관리합니다.
2. **추출/변환 단계 (ETL):** 하루에 한 번 혹은 실시간으로 운영 DB의 데이터를 추출하여 분석용으로 가공합니다.
3. **분석 단계 (OLAP):** 분석용 DB(DW)로 옮겨진 데이터를 반정규화하여 대시보드나 리포트를 만듭니다.

정규화의 원칙을 배우는 것도 중요하지만, 실제 현장에서는 "언제 정규화를 유지하고(OLTP), 언제 성능을 위해 반정규화를 선택할지(OLAP)"를 결정하는 능력이 모델러의 핵심 역량입니다.
