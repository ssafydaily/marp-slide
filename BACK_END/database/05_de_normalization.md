반정규화(De-normalization)는 정규화된 엔터티, 속성, 관계를 시스템의 성능 향상과 개발 및 운영의 단순화를 위해 의도적으로 중복, 통합, 분리하는 데이터 모델링 기법입니다.

쉽게 말해, "정석대로 쪼개놓으니 데이터 찾기가 너무 느리고 복잡해서, 성능을 위해 다시 합치는 것"을 의미합니다.

---

## 1. 반정규화가 필요한 이유 (Why?)

정규화를 거치면 테이블이 잘게 쪼개집니다. 이때 데이터를 조회하려면 여러 테이블을 합치는 **조인(Join)** 연산이 필수적인데, 데이터가 수백만 건 이상으로 많아지면 조인 비용이 급격히 증가하여 시스템이 느려집니다. 이를 해결하기 위해 반정규화를 수행합니다.

* **조회 성능 향상:** 조인 횟수를 줄여 응답 속도를 높임
* **관리 효율성:** 복잡한 SQL 쿼리를 단순화함

---

## 2. 주요 기법과 예시

### ① 테이블 통합 (중복 속성 추가)

가장 흔한 방법으로, 조인을 피하기 위해 다른 테이블의 컬럼을 복사해오는 방식입니다.

**[정규화된 상태]**

* **주문 테이블:** `[주문번호(PK), 학번(FK), 주문일자]`
* **학생 테이블:** `[학번(PK), 이름, 전공]`

> **상황:** 쇼핑몰 관리자가 주문 목록을 볼 때마다 주문한 학생의 '이름'을 항상 같이 보고 싶어 합니다. 매번 두 테이블을 조인해야 하므로 속도가 느려집니다.

**[반정규화 적용]**

* **주문 테이블:** `[주문번호(PK), 학번(FK), 학생이름, 주문일자]`
* 이렇게 `학생이름`을 주문 테이블에 중복해서 넣어두면, 학생 테이블을 조회하지 않고도 바로 이름을 확인할 수 있습니다.

---

### ② 파생 컬럼 추가 (계산된 값 미리 저장)

매번 계산이 필요한 값을 아예 컬럼으로 만들어 저장하는 방식입니다.

**[정규화된 상태]**

* **주문 테이블:** `[주문번호(PK), 주문일자]`
* **주문상세 테이블:** `[주문번호(FK), 상품ID, 수량, 단가]`

> **상황:** "총 주문 금액"을 알기 위해 매번 `수량 * 단가`를 합산(SUM)해야 합니다. 데이터가 많으면 통계 속도가 매우 느려집니다.

**[반정규화 적용]**

* **주문 테이블:** `[주문번호(PK), 주문일자, 총주문금액]`
* 주문이 발생할 때 미리 합계를 계산해서 저장해 둡니다. 조회할 때는 계산 없이 컬럼 값만 읽으면 됩니다.

---

### ③ 테이블 분할 (수직/수평 분할)

데이터가 너무 많거나 컬럼이 너무 많을 때 테이블을 쪼개는 방식입니다.

* **수평 분할 (Sharding/Partitioning):** 날짜별로 테이블을 나눕니다. (예: `2025년_매출`, `2026년_매출`)
* **수직 분할:** 자주 사용되는 컬럼과 거의 쓰이지 않는 큰 용량의 컬럼(예: 상세 설명, 첨부 파일 이미지 등)을 분리합니다.

---

## 3. 반정규화 시 주의사항 (Trade-off)

반정규화는 공짜가 아닙니다. 반드시 다음의 위험을 감수해야 합니다.

1. **데이터 무결성 위험:** 학생의 이름이 바뀌었을 때, `학생 테이블`만 수정하고 `주문 테이블`에 복사된 이름을 수정하지 않으면 데이터가 서로 달라지는 불일치가 발생합니다.
2. **저장 공간 증가:** 동일한 데이터를 여러 곳에 저장하므로 용량을 더 많이 차지합니다.
3. **쓰기 성능 저하:** 데이터를 입력하거나 수정할 때 여러 곳을 동시에 업데이트해야 하므로 수정 속도는 느려질 수 있습니다.

---

## 💡 결론: 언제 해야 할까?

반정규화는 설계의 첫 단계에서 하는 것이 아니라, **정규화를 완벽히 마친 후**에 수행합니다.

1. 먼저 **정규화**를 통해 논리적으로 완벽한 모델을 만듭니다.
2. 그다음 성능 테스트(Profiling)를 통해 **조회 병목 현상**을 확인합니다.
3. 인덱스나 캐시로 해결이 안 될 때, 최후의 수단으로 **반정규화**를 고려합니다.

학생들에게는 "정규화는 데이터의 '정직함'을 지키는 것이고, 반정규화는 시스템의 '속도'를 위해 정직함과 조금 타협하는 것"이라고 설명해 주시면 좋습니다.
