## 1. 이행적 함수 종속이란?

수학의 삼단논법과 같은 논리입니다. **A를 알면 B를 알고, B를 알면 C를 알 수 있는 관계**를 말합니다. 결과적으로 A를 통해 C를 알게 되는 '이행(Transitive)' 관계가 형성됩니다.

* **논리 구조:** $A \rightarrow B$ 이고 $B \rightarrow C$ 이면, $A \rightarrow C$ 가 성립한다.
* **핵심:** 여기서 $B$는 기본키가 아닌 **일반 속성**입니다. 즉, 기본키가 아닌 속성 간에 종속성이 발생하는 현상입니다.

---

## 2. 왜 문제가 되는가? (이상 현상)

기본키 $A$에 직접 종속되어야 할 데이터가, 중간 매개체인 $B$를 거쳐서 관리되기 때문에 데이터의 **종속 관계가 복잡해지고 중복**이 발생합니다.

* **수정 이상:** $B$의 정보가 바뀌면, $B$와 연결된 $C$의 값들을 모든 행에서 찾아 수정해야 합니다. (예: 소속 부서명이 바뀌면 해당 부서원 전체의 부서 위치 정보를 수정해야 함)
* **삭제 이상:** 특정 데이터(A)를 삭제할 때, 그와 연결된 $B$와 $C$의 관계 정보까지 원치 않게 사라집니다. (예: 사원이 퇴사해서 삭제했는데, 그 부서의 위치 정보까지 사라짐)
* **삽입 이상:** 아직 관계를 맺을 $A$가 없는 상태에서 $B$와 $C$의 정보만 미리 저장할 수 없습니다.

---

## 3. 식별 및 해결 방법 (예시 중심)

가장 이해하기 쉬운 **[사원]** 테이블 예시로 살펴보겠습니다.

### **① 식별 단계 (Identification)**

현재 테이블 구조가 다음과 같다고 가정합시다.

* **사원 테이블:** `[사원번호(PK), 이름, 부서코드, 부서명, 부서위치]`

여기서 종속 관계를 분석해 보면:

1. `사원번호` $\rightarrow$ `부서코드` (사원별로 소속 부서가 정해짐)
2. `부서코드` $\rightarrow$ `부서명`, `부서위치` (부서코드만 알면 부서 이름과 위치를 알 수 있음)
3. 결과적으로 `사원번호` $\rightarrow$ `부서명`, `부서위치`라는 **이행적 종속**이 발생!

> **질문:** "부서위치는 사원 개인의 고유 속성인가요, 아니면 부서의 속성인가요?"
> **답변:** 부서의 속성입니다. 하지만 사원 테이블에 같이 있어서 문제가 생깁니다.

---------------

### **② 해결 단계: 제3정규화 (3NF)**

해결책은 **이행 관계의 고리를 끊어내는 것**입니다. 중간 매개체가 되는 속성을 기준으로 테이블을 분리합니다.

1. 이행적 종속을 일으키는 `부서코드`, `부서명`, `부서위치`를 별도의 테이블로 분리합니다.
2. 기존 테이블에는 `부서코드`만 남겨서 **외래키(FK)** 역할을 하게 합니다.

**[분리 후 결과]**

* **사원 테이블:** `[사원번호(PK), 이름, 부서코드(FK)]`
* **부서 테이블:** `[부서코드(PK), 부서명, 부서위치]`

---

## 4. 요약 정리

| 구분 | 내용 |
| --- | --- |
| **핵심 논리** | 기본키 $\rightarrow$ 일반 속성1 $\rightarrow$ 일반 속성2 |
| **식별 포인트** | **"기본키가 아닌 속성들끼리 서로 결정하고 있는가?"**를 확인 |
| **해결 방법** | **제3정규화(3NF)** 수행 (중간 결정자를 PK로 하는 새 테이블 생성) |
| **비유** | "친구의 친구를 아는 것". 친구(부서코드)가 이사가면 그 집 주소(부서위치)를 내 연락처(사원 테이블)에서 직접 관리하지 말고, 친구의 주소록(부서 테이블)을 따로 만들어 관리하라는 뜻입니다. |

이행적 함수 종속을 제거하면 데이터 모델은 훨씬 더 **유연**해지고, 부서 정보가 바뀌더라도 사원 데이터를 건드릴 필요가 없는 **독립성**을 갖게 됩니다. 수업 자료에 이 사원-부서 예시를 넣으시면 학생들이 가장 직관적으로 이해할 것입니다!
